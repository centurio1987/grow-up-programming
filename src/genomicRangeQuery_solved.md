# genomicRangeQuery 풀이 캔버스

## 성능 목표 예측

| 항목 | 값 |
|------|---|
| N (DNA 길이) | 최대 100,000 |
| M (쿼리 수) | 최대 50,000 |
| 브루트포스 O(N·M) | ~5×10⁹ 연산 → 초과 |
| 목표 복잡도 | O(N + M) |

CPU 기준 10⁸ 연산/초 가정 시, O(N+M) ≈ 150,000 → 충분히 1초 이내 처리 가능.

---

## 목표 함수

$$\text{answer}[K] = \min_{P[K] \leq i \leq Q[K]} \text{impact}(S[i])$$

$$\text{impact}(c) = \begin{cases} 1 & c = \text{A} \\ 2 & c = \text{C} \\ 3 & c = \text{G} \\ 4 & c = \text{T} \end{cases}$$

---

## 로직

### 도메인 컨셉과 조건 정의

최솟값을 직접 구하는 대신, **충격 지수가 낮은 순서대로 구간 내 존재 여부를 판단**하면 된다.

A=1 < C=2 < G=3 < T=4 이므로, A가 존재하면 답은 무조건 1이다.

구간 내 특정 문자 c의 등장 횟수를 O(1)에 계산하기 위해 prefix sum을 사용한다:

$$\text{prefix}_c[i] = \#\{j \mid 0 \leq j < i,\; S[j] = c\}$$

구간 $[p, q]$ 내 c의 등장 횟수:

$$\text{count}_c(p, q) = \text{prefix}_c[q+1] - \text{prefix}_c[p]$$

T는 별도 prefix sum이 불필요하다. A/C/G 모두 0이면 T만 존재하므로 자동으로 4를 반환한다.

### 입력-도메인-출력을 포함한 수도코드

```
입력: S (길이 N), P, Q (길이 M)
출력: answer (길이 M)

[전처리 — O(N)]
prefixA[0] = prefixC[0] = prefixG[0] = 0
for i in 0..N-1:
  prefixA[i+1] = prefixA[i] + (S[i] == 'A' ? 1 : 0)
  prefixC[i+1] = prefixC[i] + (S[i] == 'C' ? 1 : 0)
  prefixG[i+1] = prefixG[i] + (S[i] == 'G' ? 1 : 0)

[쿼리 처리 — O(M)]
for K in 0..M-1:
  p = P[K], q = Q[K]
  if prefixA[q+1] - prefixA[p] > 0:  answer[K] = 1
  elif prefixC[q+1] - prefixC[p] > 0: answer[K] = 2
  elif prefixG[q+1] - prefixG[p] > 0: answer[K] = 3
  else:                                answer[K] = 4
```

### 직접 수행으로 바운더리 값 검증

S = "CAGCCTA" (N=7, 인덱스 0~6):

```
인덱스:   0   1   2   3   4   5   6
문자:     C   A   G   C   C   T   A
```

prefix 배열 (길이 8):

| i | prefixA | prefixC | prefixG |
|---|---------|---------|---------|
| 0 | 0 | 0 | 0 |
| 1 | 0 | 1 | 0 |
| 2 | 1 | 1 | 0 |
| 3 | 1 | 1 | 1 |
| 4 | 1 | 2 | 1 |
| 5 | 1 | 3 | 1 |
| 6 | 1 | 3 | 1 |
| 7 | 2 | 3 | 1 |

쿼리 검증:

| K | p | q | A count | C count | G count | answer |
|---|---|---|---------|---------|---------|--------|
| 0 | 2 | 4 | 1−1=0 | 3−1=2>0 | — | **2** ✓ |
| 1 | 5 | 5 | 1−1=0 | 3−3=0 | 1−1=0 | **4** ✓ |
| 2 | 0 | 6 | 2−0=2>0 | — | — | **1** ✓ |

바운더리:

- `S="A"`, `[0,0]`: prefixA[1]−prefixA[0]=1>0 → **1** ✓
- `S="T"`, `[0,0]`: A=0, C=0, G=0 → **4** ✓
- `S="GCAT"`, 단일 쿼리 [0,0],[1,1],[2,2],[3,3]: G→3, C→2, A→1, T→4 → **[3,2,1,4]** ✓

## 성능 최적화 로직

### 도메인 컨셉과 조건 정의

브루트포스(O(N·M))의 병목은 각 쿼리마다 구간을 직접 순회하는 것이다.

**전처리 + 쿼리 분리 패턴**: prefix sum을 O(N)에 한 번만 구성하면, 이후 각 쿼리를 O(1)에 처리할 수 있다.

$$\text{총 복잡도} = O(N) + O(M) = O(N + M)$$

T를 포함한 4개 배열을 만들 필요 없이, **A/C/G 3개만으로 충분**:

- A/C/G 중 하나라도 구간 내에 있으면 → 해당 충격 지수 반환
- 셋 다 없으면 → T만 있으므로 4 반환

### 입력-도메인-출력을 포함한 수도코드

브루트포스 대비 변경점:
- 이중 루프 제거
- 구간별 문자 검색 → prefix sum O(1) 조회로 교체

### 직접 수행으로 바운더리 값 검증

N=100,000, M=50,000 성능 테스트:
- 전처리: 100,000번 반복 × 3 문자 체크 = 300,000 연산
- 쿼리: 50,000번 × 최대 3번 비교 = 150,000 연산
- 총 ~450,000 연산 → 1000ms 이내 충분히 처리 가능

# 캔버스 변경의 원인이 된 사고의 변경 이력

1. **초기 접근 (브루트포스)**: 각 쿼리마다 S[p..q]를 직접 순회 → O(N·M), N=M=최대 기준 50억 연산으로 시간 초과.

2. **전환 (prefix sum)**: "구간 내 최솟값"을 직접 구하는 대신, "가장 낮은 충격 지수 문자가 구간 내에 존재하는가?"로 문제를 변환. 존재 여부는 prefix sum 차이로 O(1) 판단 가능 → O(N+M)으로 해결.

3. **최적화 (T 생략)**: A/C/G 세 문자의 prefix sum만으로 충분. T는 나머지 경우에 해당하므로 별도 배열 불필요 → 공간과 전처리 연산 25% 절감.
