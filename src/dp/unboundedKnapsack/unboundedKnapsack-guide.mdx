# unboundedKnapsack — 동전 교환 최소 개수 해설

## 성능 목표 예측

| 제약 | 값 |
|------|-----|
| 동전 종류 $n$ | $1 \leq n \leq 100$ |
| 목표 금액 $amount$ | $0 \leq amount \leq 10^4$ |
| 동전 액면가 $c_i$ | $1 \leq c_i \leq 10^4$ |

**Naive 접근의 복잡도 분석**

각 동전을 사용할 수 있는 최대 횟수는 $\lfloor amount / c_i \rfloor$이다. 극단적으로 $c_i = 1$이면 동전 하나를 최대 $amount$번 쓸 수 있다. 모든 동전의 사용 횟수 조합을 완전 탐색하면 최악의 경우 $O(amount^n)$ 가지이다. $amount = 10^4$, $n = 100$이면 불가능한 규모다.

재귀 + 메모이제이션 없이 구현하면, 동일한 금액에 대한 계산이 지수적으로 반복된다.

**목표 복잡도**

$$O(n \cdot amount) = O(100 \times 10^4) = O(10^6)$$

상태 수가 $O(amount)$이고 각 상태에서 $n$가지 동전을 검토하므로 목표를 달성한다. $O(10^6)$은 현대 CPU에서 수 밀리초 안에 처리된다.

**공간 복잡도**

$$O(amount)$$

1D DP 배열 하나만 필요하다. $amount \leq 10^4$이므로 배열 크기가 최대 10,001 원소다.

---

## 목표 함수

```ts
function unboundedKnapsack(coins: number[], amount: number): number
```

| 파라미터 | 의미 | 제약 |
|----------|------|------|
| `coins` | 동전 액면가 배열 | $1 \leq n \leq 100$, $1 \leq c_i \leq 10^4$ |
| `amount` | 만들어야 할 목표 금액 | $0 \leq amount \leq 10^4$ |
| 반환값 | 최소 동전 개수; 불가능하면 $-1$ | 정수 |

**엣지케이스 목록**

| 입력 | 기대 출력 | 이유 |
|------|-----------|------|
| `coins=[1,2,5], amount=0` | `0` | 금액 0을 만드는 데 동전 0개 필요 |
| `coins=[2], amount=3` | `-1` | 짝수 동전으로 홀수 만들기 불가 |
| `coins=[1], amount=10000` | `10000` | 1짜리 동전 10000개 |
| `coins=[10000], amount=10000` | `1` | 정확히 맞는 동전 한 개 |

## 핵심 아이디어

**핵심 아이디어**: "금액을 작은 값부터 채우면, 같은 동전을 여러 번 쓰는 경우가 자연스럽게 누적된다."

동전 교환 최소 개수 문제는 "금액 $x$를 만드는 최소 동전 수"라는 1D 상태만으로 충분하다. 금액을 1부터 `amount`까지 오름차순으로 채울 때 `dp[x-c]`는 이미 최적값으로 확정된 상태이므로 `dp[x] = min(dp[x], dp[x-c]+1)`이 올바르게 적용된다. 동전을 무한 사용할 수 있으므로 오름차순 갱신이 자연스럽다는 점이 0/1 배낭(내림차순)과의 핵심 차이다.

**풀이 구조**
1. `dp[0] = 0`, `dp[1..amount] = +∞`로 초기화한다.
2. 금액 `x = 1`부터 `amount`까지 바깥 루프를 돈다.
3. 각 동전 `c`에 대해 `c <= x`이고 `dp[x-c] != +∞`이면 `dp[x] = min(dp[x], dp[x-c]+1)`로 갱신한다.
4. `dp[amount] == +∞`이면 `-1`, 아니면 `dp[amount]`를 반환한다.

**조건**: 각 동전을 무제한으로 반복 사용할 수 있고(언바운드), 정확히 `amount`를 만드는 최소 동전 개수를 구하는 문제에 적용 가능하다.

**대표 예시**: `coins=[1,5,6], amount=11`
그리디로는 6+5=2개이지만, DP는 `dp[6]=1`, `dp[10]=2`, `dp[11]=min(dp[10]+1, dp[6]+dp[5]+1...)` 과정에서 `dp[11] = dp[5]+dp[6] → 2`가 아니라 `dp[6]+1=2`와 `dp[10]+1=3`을 비교해 최적 2개(6+5)를 정확히 찾는다.

**언제 쓰나**
"주어진 코인/타일/점프 단위를 무제한 사용해 목표값을 정확히 만드는 최소 횟수"를 구하는 문제에서 이 언바운드 DP 패턴을 사용한다.

### 원형 아이디어와 naive 접근

금액 $amount$를 만드는 모든 동전 조합을 재귀로 탐색하고, 그 중 동전 개수가 최소인 것을 찾는다.

```
function solve(remaining):
    if remaining == 0: return 0     // 만든 경우, 동전 0개 더 필요
    if remaining < 0: return Infinity
    best = Infinity
    for c in coins:
        sub = solve(remaining - c)
        if sub != Infinity:
            best = min(best, sub + 1)
    return best
```

이 재귀는 `solve(remaining)` 상태가 `solve(remaining - c)`에서 유도되는데, $remaining$이 같은 상태가 여러 경로에서 반복 등장한다. 예를 들어 `solve(10)`은 동전 3과 7을 통해 각각 `solve(7)`과 `solve(3)`을 호출하고, 이 둘도 다시 서로를 호출할 수 있다. 메모이제이션 없이는 지수적 재계산이 발생한다.

### 어떤 관찰이 돌파구가 되는가

- **관찰 1 — 상태가 1D이다**: 금액 $x$를 만드는 최소 동전 수는 $x$만으로 결정된다. 동전 조합의 순서나 이력은 중요하지 않다. 따라서 1D 배열 $dp[x]$로 모든 상태를 표현할 수 있다.
- **관찰 2 — 최적 부분구조**: 금액 $x$를 최소 동전으로 만들 때, 마지막에 사용한 동전 $c$를 빼면 $dp[x] = dp[x - c] + 1$이 성립한다. $dp[x - c]$가 이미 최적값이라면, 여기에 1을 더한 것이 $x$를 이 동전으로 끝낸 최솟값이다.
- **관찰 3 — 무한 사용과 루프 순서**: 동전을 무한히 사용할 수 있으므로, 금액을 작은 값부터 채워도 같은 동전을 여러 번 참조하는 것이 자연스럽다. 이는 0/1 배낭과 반대 방향이다.

### 관찰을 형식화: 상태/구조 정의

$$dp[x] = \text{"금액 } x \text{를 만드는 데 필요한 최소 동전 수"}$$

- 달성 불가능한 금액: $dp[x] = +\infty$ (또는 충분히 큰 수)
- 달성 가능한 금액: $dp[x] = $ 유한한 최솟값

초기 조건:

$$dp[0] = 0, \quad dp[x] = +\infty \;\; (x > 0)$$

$dp[0] = 0$의 의미: 금액 0을 만드는 데 동전이 0개 필요하다(빈 선택).

왜 이 1D 정의로 충분한가? 2D로 정의하면 `dp[i][x]` = "$i$번째 동전까지만 사용해 금액 $x$를 만드는 최소 동전 수"가 되는데, 동전을 무한히 사용할 수 있으므로 동전 종류 순서가 결과에 영향을 주지 않는다. 어떤 동전이든 금액이 허용한다면 언제든 사용할 수 있으므로 "어느 동전까지 고려했는가"를 별도 차원으로 유지할 필요가 없다.

### 점화식 또는 핵심 연산

금액 $x$ ($1 \leq x \leq amount$)에 대해, 모든 동전 $c \in coins$ ($c \leq x$)를 시도한다.

$$dp[x] = \min_{c \in coins,\; c \leq x} \bigl(dp[x - c] + 1\bigr)$$

각 항의 의미:
- $dp[x - c]$: 금액 $x - c$를 이미 만들었을 때의 최소 동전 수
- $+1$: 동전 $c$ 하나를 추가
- $\min$: 가능한 모든 마지막 동전 중 최소를 선택

외부 루프를 금액 $x$로, 내부 루프를 동전 $c$로 두는 순서를 사용한다. 이 순서에서 $dp[x - c]$는 이미 계산된 값이다($x - c < x$이므로).

### 정당성 — 왜 이것이 옳은가

귀납으로 검증한다. $dp[0] = 0$은 자명하다. $dp[0], dp[1], \ldots, dp[x-1]$이 모두 올바르다고 가정하면, $dp[x]$는 "금액 $x$를 만드는 모든 방법 중 마지막 동전 $c$를 기준으로 $dp[x-c] + 1$의 최솟값"이다. $dp[x-c]$는 귀납 가정에 의해 올바르고, 가능한 모든 $c$에 대해 $\min$을 취하므로 $dp[x]$도 올바르다.

무한 사용이 올바르게 반영되는 이유: $x$를 작은 값부터 채우므로 $dp[c]$가 이미 채워진 상태에서 $dp[2c] = dp[c] + 1$, $dp[3c] = dp[2c] + 1 = dp[c] + 2$ 등이 자연스럽게 계산된다. 같은 동전을 여러 번 사용하는 경우가 정확히 처리된다.

특수 케이스:
- 어떤 동전도 $x$를 만들 수 없으면: 모든 $c$에 대해 $dp[x-c] = +\infty$이므로 $dp[x]$가 $+\infty$로 유지된다. 최종적으로 $dp[amount] = +\infty$이면 $-1$을 반환한다.
- $dp[x-c] = +\infty$인 경우 $+\infty + 1$이 오버플로를 일으킬 수 있으므로, $dp[x-c] \neq +\infty$를 조건으로 확인한 후 갱신해야 한다.

### 구현 디테일과 최적화

**루프 순서**: 금액을 바깥 루프, 동전을 안쪽 루프로 두는 방식이 직관적이다. 반대로 동전을 바깥 루프, 금액을 안쪽 루프(오름차순)로 두어도 동일한 결과가 나온다. 이는 0/1 배낭과의 중요한 차이점이다.

**coinChangeWays와의 루프 순서 비교**:
- `coinChangeWays`(조합 수): 동전 바깥, 금액 안쪽 → 순서가 중요(조합)
- `unboundedKnapsack`(최소 개수): 금액 바깥, 동전 안쪽 → 순서가 결과에 무관(최솟값)

이 차이가 생기는 이유: 최솟값을 구할 때는 어떤 순서로 갱신해도 $\min$이 동일한 최솟값으로 수렴하지만, 카운팅에서는 중복 카운트가 발생하지 않으려면 루프 순서가 중요하다.

**흔한 함정**:
- `+Infinity`인 `dp[x-c]`에 1을 더하면 오버플로가 발생한다. 반드시 `dp[x-c] !== Infinity` 또는 `dp[x-c] < amount + 1` 등의 조건 검사를 추가한다.
- 최종 반환값에서 `dp[amount] === Infinity`이면 $-1$을 반환하는 것을 빠뜨리기 쉽다.

## 수도 코드와 Activity Diagram

### 의사코드

```
function unboundedKnapsack(coins, amount):
    dp = 크기 (amount+1)인 배열, +Infinity로 초기화
    dp[0] = 0
    // 불변식: dp[x]는 "금액 x를 만드는 최소 동전 수" (불가능하면 +Infinity)

    for x = 1 to amount:               // 금액을 작은 값부터 채움
        for c in coins:
            if c <= x and dp[x - c] != +Infinity:
                dp[x] = min(dp[x], dp[x - c] + 1)
        // 루프 불변식: dp[1..x]가 모두 최적값으로 확정됨

    return dp[amount] == +Infinity ? -1 : dp[amount]
```

**핵심 불변식**: 바깥 루프에서 금액 $x$의 처리가 완료된 시점에, $dp[x]$는 "코인 집합 전체를 무제한 사용해 정확히 금액 $x$를 만드는 최소 동전 수"이다. $x$보다 작은 값들은 이미 확정되어 있으므로 $dp[x-c]$ 참조가 항상 올바르다.
