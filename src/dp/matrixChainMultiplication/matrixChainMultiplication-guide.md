# matrixChainMultiplication — 행렬 체인 곱셈 해설

## 성능 목표 예측

| 제약 | 값 |
|------|-----|
| 행렬 수 $n$ | $1 \leq n \leq 100$ |
| 각 차원 $dims[i]$ | $1 \leq dims[i] \leq 500$ |
| `dims` 배열 길이 | $n + 1$ ($\leq 101$) |

**Naive 접근의 복잡도 분석**

$n$개의 행렬을 곱하는 모든 괄호 배치를 완전 탐색하면, 카탈란 수(Catalan number) $C_{n-1}$에 비례하는 경우의 수가 생긴다.

$$C_{n-1} = \frac{1}{n}\binom{2(n-1)}{n-1} \sim \frac{4^{n-1}}{n^{3/2}\sqrt{\pi}}$$

$n = 100$이면 $C_{99} \approx 4^{99} / \text{poly}(99) \approx 10^{57}$가지이다. 이는 완전 탐색으로 절대 해결할 수 없는 규모다.

**목표 복잡도**

$$O(n^3) = O(100^3) = O(10^6)$$

구간 DP는 상태 수 $O(n^2)$, 각 상태를 채우는 데 $O(n)$ 작업을 하므로 전체 $O(n^3)$이다. $n = 100$에서 $10^6$ 연산이므로 충분히 빠르다.

**공간 복잡도**

$$O(n^2) = O(10^4)$$

$n \times n$ 2D DP 테이블 하나만 필요하다. 롤링 배열 최적화가 불가능한 구조이므로 2D를 그대로 사용한다.

## 목표 함수

```ts
function matrixChainMultiplication(dims: number[]): number
```

| 파라미터 | 의미 | 제약 |
|----------|------|------|
| `dims` | 행렬 차원 배열; $A_i$의 크기는 $dims[i-1] \times dims[i]$ | 길이 $n+1$, $1 \leq n \leq 100$ |
| 반환값 | 최소 스칼라 곱셈 횟수 | $\geq 0$ |

**엣지케이스 목록**

| 입력 | 기대 출력 | 이유 |
|------|-----------|------|
| `dims=[10, 20]` | `0` | 행렬이 1개이므로 곱셈 없음 |
| `dims=[2, 3, 4]` | `24` | $A_1(2\times3) \cdot A_2(3\times4)$: $2\times3\times4=24$, 순서 하나뿐 |
| `dims=[10, 100, 5, 50]` | `7500` | 최적: $((A_1 A_2) A_3)$, 비용 $10\cdot100\cdot5 + 10\cdot5\cdot50 = 5000+2500=7500$ |
| `dims=[500]*101` | (큰 수) | 최대 입력; 오버플로 없는지 확인 필요 |

## 핵심 아이디어

**핵심 아이디어**: "어디서 마지막으로 두 덩어리로 쪼갤지를 결정하면, 나머지는 더 작은 구간의 최적값으로 표현된다."

행렬을 곱하는 순서(괄호 배치)에 따라 스칼라 곱셈 횟수가 크게 달라진다. 핵심 관찰은 어떤 괄호 배치든 반드시 마지막으로 두 부분을 합치는 분할점 $k$가 존재하고, 그 두 부분 각각도 최적으로 묶여야 전체가 최적이라는 것이다. 이 최적 부분구조 덕분에 구간을 짧은 것부터 긴 것 순서로 채우는 구간 DP가 가능하다.

**풀이 구조**
1. `dp[i][i] = 0`으로 초기화한다(행렬 하나는 곱셈 비용 없음).
2. 구간 길이 2부터 n까지 바깥 루프를 돈다.
3. 각 구간 `[i, j]`에 대해 분할점 `k`를 `i`부터 `j-1`까지 시도한다.
4. `dp[i][j] = min(dp[i][k] + dp[k+1][j] + dims[i-1]*dims[k]*dims[j])`로 갱신한다.
5. `dp[1][n]`을 반환한다.

**조건**: 연속된 원소들을 순서대로 결합해야 하고, 결합 순서(괄호 배치)에 따라 비용이 달라지는 문제에 적용 가능하다. 최적 부분구조가 성립해야 한다.

**대표 예시**: `dims=[10, 100, 5, 50]` (행렬 3개: $A_1(10\times100)$, $A_2(100\times5)$, $A_3(5\times50)$)
`((A_1 A_2) A_3)` 방식: $10\times100\times5 + 10\times5\times50 = 5000+2500 = 7500$,
`(A_1 (A_2 A_3))` 방식: $100\times5\times50 + 10\times100\times50 = 25000+50000 = 75000$.
구간 DP가 자동으로 7500이 최적임을 찾아낸다.

**언제 쓰나**
"연속된 N개의 원소를 임의 순서로 결합할 때 총 비용을 최소화하라"는 형태(행렬 곱, 문자열 연결 비용, 돌 합치기 등)에서 이 구간 DP 패턴을 사용한다.

### 원형 아이디어와 naive 접근

행렬 체인 $A_1 A_2 \cdots A_n$을 곱하는 모든 괄호 배치를 재귀적으로 탐색한다.

```
function minCost(i, j):
    if i == j: return 0     // 행렬 하나는 비용 없음
    best = Infinity
    for k = i to j-1:      // 어디서 마지막으로 분할할지 시도
        cost = minCost(i, k) + minCost(k+1, j) + dims[i-1] * dims[k] * dims[j]
        best = min(best, cost)
    return best
```

이 재귀의 문제는 중복 부분문제다. `minCost(2, 5)`는 구간 [1,6]을 처리하는 여러 경로에서 반복 등장하지만, 매번 다시 계산한다. 전체 시간복잡도는 카탈란 수에 비례하는 지수 시간이 된다.

### 어떤 관찰이 돌파구가 되는가

- **관찰 1 — 최적 부분구조**: 구간 $[i, j]$를 최적으로 괄호 묶는 방법에서, 어딘가 위치 $k$에서 마지막으로 두 부분으로 나뉜다. 이때 $[i, k]$와 $[k+1, j]$도 각자 최적으로 괄호 묶여야 한다. 그렇지 않으면 더 좋은 분할로 교체해 전체를 개선할 수 있기 때문이다. 이 최적 부분구조가 DP를 적용할 수 있는 근거다.
- **관찰 2 — 중복 부분문제**: 구간 $[i, j]$의 최솟값을 구하는 부분 문제는 $O(n^2)$가지밖에 없다. 메모이제이션으로 각 구간을 한 번만 계산한다.
- **관찰 3 — 구간 의존성의 방향**: 큰 구간의 최솟값은 더 작은 구간의 최솟값에 의존한다. 따라서 구간 길이가 짧은 것부터 긴 것 순서로 채우면 필요한 값이 항상 먼저 준비된다.

### 관찰을 형식화: 상태/구조 정의

$$dp[i][j] = \text{"행렬 } A_i A_{i+1} \cdots A_j \text{를 곱하는 최소 스칼라 곱셈 횟수"} \quad (1 \leq i \leq j \leq n)$$

초기 조건:

$$dp[i][i] = 0 \quad \forall i$$

행렬 하나를 곱하는 데는 곱셈이 필요 없다.

왜 이 2D 정의가 필요한가? 상태에서 구간의 시작과 끝을 모두 표현해야 하는 이유는, 분할점 $k$를 기준으로 $[i, k]$와 $[k+1, j]$라는 두 하위 구간이 필요하기 때문이다. 1D 상태는 이 구조를 표현하지 못한다.

### 점화식 또는 핵심 연산

구간 $[i, j]$ ($i < j$)에 대해, 분할점 $k$ ($i \leq k < j$)를 시도한다.

$$dp[i][j] = \min_{i \leq k < j} \bigl( dp[i][k] + dp[k+1][j] + dims[i-1] \cdot dims[k] \cdot dims[j] \bigr)$$

각 항의 의미:
- $dp[i][k]$: 왼쪽 부분 $A_i \cdots A_k$를 최적으로 곱하는 비용
- $dp[k+1][j]$: 오른쪽 부분 $A_{k+1} \cdots A_j$를 최적으로 곱하는 비용
- $dims[i-1] \cdot dims[k] \cdot dims[j]$: 두 결과 행렬 $(dims[i-1] \times dims[k])$와 $(dims[k] \times dims[j])$를 곱하는 비용

비용 계산의 근거: $A_i \cdots A_k$의 결과 행렬은 $dims[i-1] \times dims[k]$ 크기이고, $A_{k+1} \cdots A_j$의 결과 행렬은 $dims[k] \times dims[j]$ 크기이다. 두 행렬 $(p \times q)$와 $(q \times r)$의 곱에서 스칼라 곱셈 횟수는 $p \times q \times r$이다.

결과: $dp[1][n]$

### 정당성 — 왜 이것이 옳은가

귀납으로 검증한다. 구간 길이 $len = 1$일 때 $dp[i][i] = 0$이 자명하게 옳다. 구간 길이 $< len$의 모든 $dp[i][j]$가 올바르다고 가정하면, 구간 길이 $= len$인 $dp[i][j]$를 계산할 때 참조하는 $dp[i][k]$와 $dp[k+1][j]$의 길이는 모두 $< len$이므로 귀납 가정에 의해 올바르다. $\min$을 통해 모든 분할점을 시도하므로 최적값이 보장된다.

특수 케이스:
- $n = 1$ (행렬 하나): `dims`의 길이가 2이므로 $dp[1][1] = 0$이 반환된다.
- 모든 차원이 같을 경우: 여전히 분할점에 따라 비용이 달라질 수 있으므로 min 탐색이 필요하다.

### 구현 디테일과 최적화

**루프 순서 — 구간 길이 우선**: `for len = 2 to n` → `for i = 1 to n-len+1` → `j = i + len - 1` → `for k = i to j-1`. 이 순서가 의존성을 올바르게 해결한다. `for i = 1 to n` → `for j = i+1 to n` 순서(대각선 방향)로도 동일하게 가능하다.

**인덱스 관계**: 행렬은 1-indexed($A_1 \ldots A_n$), `dims`는 0-indexed다. $A_i$의 크기는 $dims[i-1] \times dims[i]$이다. 구간 $[i, j]$의 결과 행렬 크기는 $dims[i-1] \times dims[j]$이다.

**흔한 함정**:
- $dp[i][j]$의 초기값을 $-\infty$나 0으로 설정하면 $\min$ 연산이 깨진다. 반드시 $+\infty$로 초기화한다.
- 분할점 $k$의 범위를 $i \leq k < j$로 설정하지 않으면(예: $k = j$를 포함하면) $dp[j+1][j]$처럼 비어 있는 구간을 참조하게 된다.
- JavaScript에서 `Infinity`를 사용하면 `Infinity + Infinity = Infinity`이므로 문제없지만, 정수 타입의 경우 오버플로에 주의해야 한다. $n = 100$, $dims[i] = 500$이면 최대 비용은 약 $500^3 \times 100 \approx 1.25 \times 10^{10}$으로 32비트 정수 범위를 초과할 수 있다.

## 수도 코드와 Activity Diagram

### 의사코드

```
function matrixChainMultiplication(dims):
    n = dims.length - 1     // 행렬 수
    if n <= 1: return 0

    // dp[i][j]: 행렬 i..j를 곱하는 최소 스칼라 곱셈 횟수 (1-indexed)
    dp = n × n 2D 배열, 0으로 초기화
    // 불변식: dp[i][i] = 0 (초기화 시 이미 0)

    for len = 2 to n:           // 구간 길이 (짧은 것부터)
        for i = 1 to n - len + 1:   // 구간 시작
            j = i + len - 1         // 구간 끝
            dp[i][j] = +Infinity
            for k = i to j - 1:    // 분할점
                cost = dp[i][k] + dp[k+1][j] + dims[i-1] * dims[k] * dims[j]
                // 불변식: cost는 A_i..A_k와 A_{k+1}..A_j를 합치는 총 비용
                if cost < dp[i][j]:
                    dp[i][j] = cost

    return dp[1][n]
```
