# coinChangeWays — 동전으로 금액 만드는 경우의 수 해설

## 성능 목표 예측

| 제약 | 값 |
|------|-----|
| 동전 종류 $n$ | $\leq 100$ |
| 목표 금액 $amount$ | $\leq 10^4$ |
| 동전 액면가 $c_i$ | $1 \leq c_i \leq 10^4$ |

**Naive 접근의 복잡도 분석**

가장 단순한 접근은 재귀적 완전 탐색이다. 금액 $amount$를 만들기 위해 $n$가지 동전 각각을 0회, 1회, 2회, … 사용하는 모든 경우를 열거한다. 최악의 경우 $c_i = 1$이면 동전을 최대 $amount$번 쓸 수 있으므로 탐색 공간은 $O(amount^n)$이다. $amount = 10^4$, $n = 100$이면 이 값은 천문학적으로 커서 시간 초과가 된다.

**목표 복잡도**

$$O(n \cdot amount) = O(100 \times 10^4) = O(10^6)$$

이 정도면 현대 CPU가 약 $10^8$ ~ $10^9$ 연산/초를 처리하므로 충분히 통과한다.

**공간 복잡도**

- 2D 테이블 $O(n \cdot amount)$ 대신 **1D 롤링 배열** $O(amount)$를 사용한다.
- $amount \leq 10^4$이면 1D 배열 크기가 최대 10,001 원소이므로 메모리 부담이 없다.

---

## 목표 함수

```ts
function coinChangeWays(coins: number[], amount: number): number
```

| 파라미터 | 의미 | 제약 |
|----------|------|------|
| `coins` | 동전 액면가 배열 | $1 \leq n \leq 100$, $1 \leq c_i \leq 10^4$ |
| `amount` | 만들어야 할 목표 금액 | $0 \leq amount \leq 10^4$ |
| 반환값 | 순서를 구분하지 않는 조합의 수 | $\geq 0$ |

**엣지케이스 목록**

| 입력 | 기대 출력 | 이유 |
|------|-----------|------|
| `coins=[1,2], amount=0` | `1` | 빈 조합이 유일한 방법 |
| `coins=[5], amount=3` | `0` | 어떤 동전도 3을 만들 수 없음 |
| `coins=[1], amount=10000` | `1` | 동전 1짜리만으로 만드는 방법은 한 가지 |
| `coins=[1,2,5], amount=10000` | (큰 수) | 최대 입력 범위, 오버플로 주의 |

## 핵심 아이디어

### 원형 아이디어와 naive 접근

금액 $amount$를 만드는 모든 조합을 직접 열거하는 재귀를 생각해 보자.

```
function count(remaining, coinIndex):
    if remaining == 0: return 1   // 정확히 만든 경우
    if remaining < 0 or coinIndex >= n: return 0
    total = 0
    for k = 0, 1, 2, ...:        // 동전 coinIndex를 k번 사용
        if coins[coinIndex] * k > remaining: break
        total += count(remaining - coins[coinIndex]*k, coinIndex + 1)
    return total
```

문제는 **중복 부분문제**이다. 예를 들어 `coins=[1,2,3], amount=6`일 때, (1을 0번, 2를 3번 사용해 남은 금액 0)과 (1을 2번, 2를 2번 사용해 남은 금액 0) 등 많은 경로에서 같은 `(remaining=0, coinIndex=2)` 상태가 반복 등장한다. 메모이제이션 없이는 같은 계산을 지수적으로 반복하게 된다.

또한 **순서 중복** 문제도 있다. 나이브하게 구현하면 {1,2}와 {2,1}을 서로 다른 조합으로 세는 순열 카운트가 되어 버린다. 문제는 조합(순서 무관)을 요구하므로 이를 막아야 한다.

### 어떤 관찰이 돌파구가 되는가

- **관찰 1 — 중복 부분문제**: `(남은 금액 x, 허용 동전 집합)`이 동일한 상황은 여러 탐색 경로에서 반복 등장한다. 이를 테이블에 저장하면 재계산을 없앨 수 있다.
- **관찰 2 — 조합과 순열의 차이는 "동전 처리 순서"에서 온다**: 동전을 한 종류씩 고정하고 금액을 순회하면, 나중 동전은 항상 앞서 처리된 동전 "위에" 추가되는 구조가 된다. 이렇게 하면 같은 동전 집합이 순서가 달라도 딱 한 번만 셀 수 있다.
- **관찰 3 — 무한 사용 허용**: 동전은 무한히 있으므로, 같은 동전을 여러 번 더할 수 있어야 한다. 금액을 **작은 값부터** 순회하면 이미 갱신된 값을 다시 참조해 같은 동전을 여러 번 쓰는 효과를 얻는다.

### 관찰을 형식화: 상태/구조 정의

상태를 다음과 같이 정의한다.

$$dp[x] = \text{"지금까지 처리한 동전들만 사용해 금액 } x \text{를 만드는 조합의 수"}$$

초기 조건:

$$dp[0] = 1, \quad dp[x] = 0 \;\; (x > 0)$$

$dp[0] = 1$의 의미: 아무 동전도 쓰지 않고 금액 0을 만드는 방법이 정확히 1가지(빈 조합)이다.

왜 이 정의여야 하는가? 만약 상태를 `(금액, 남은 동전 종류)`의 2D로 두면 동일한 결과를 얻지만 공간이 $O(n \cdot amount)$가 된다. 동전을 바깥 루프로 처리하면 "허용 동전 집합" 차원을 암묵적으로 루프가 처리하므로 1D로 충분하다.

### 점화식 또는 핵심 연산

동전 $c$를 처리하는 단계에서, 금액 $x$($c \leq x \leq amount$)에 대해:

$$dp[x] \mathrel{+}= dp[x - c]$$

각 항의 의미:
- $dp[x]$: 동전 $c$를 고려하기 전(= 이미 처리된 동전들만)에 금액 $x$를 만드는 조합 수
- $dp[x - c]$: 동전 $c$를 한 번 더 추가했을 때 남는 금액 $x - c$를 만드는 조합 수
- $dp[x] \mathrel{+}= dp[x - c]$: "금액 $x - c$를 만들던 조합에 동전 $c$ 하나를 얹어 금액 $x$를 만드는 방법"을 누적

루프를 $x = c$부터 $amount$까지 **오름차순**으로 돌린다. 이 방향이 "동전 $c$를 여러 번 사용"을 자연스럽게 허용한다. $x = c$일 때 $dp[0]$를 참조하고, $x = 2c$일 때 이미 갱신된 $dp[c]$를 참조하므로 동전 $c$를 2번 사용하는 경우까지 셀 수 있다.

### 정당성 — 왜 이것이 옳은가

귀납적으로 증명한다. 동전 집합 $\{c_1, \ldots, c_k\}$까지 처리를 마쳤을 때 $dp[x]$가 정확히 "이 동전들로 금액 $x$를 만드는 조합의 수"라고 가정하자. 동전 $c_{k+1}$을 처리하면 $x = c_{k+1}$부터 $amount$까지 $dp[x] \mathrel{+}= dp[x - c_{k+1}]$을 수행한다. 이는 기존 조합에 $c_{k+1}$을 1개 더 얹는 모든 경우를 추가하는 것이다. $c_{k+1}$을 2번, 3번, … 쓰는 경우도 $x$가 증가하며 자동으로 누적된다.

순서 중복이 발생하지 않는 이유: 동전이 바깥 루프를 돌기 때문에, $dp[x]$가 동전 $c_1, \ldots, c_k$로 만들어진 상태에서 $c_{k+1}$을 추가한다. $c_{k+1}$보다 나중에 나오는 동전은 아직 반영되지 않았으므로, {$c_{k+1}$, 이전 동전} 순서가 아닌 {이전 동전, $c_{k+1}$} 순서만 생성된다.

특수 케이스: $amount = 0$이면 루프가 $dp[0]$을 건드리지 않으므로 $dp[0] = 1$이 그대로 반환된다. 어떤 동전도 $amount$를 만들 수 없으면 갱신이 한 번도 일어나지 않아 $dp[amount] = 0$이 반환된다.

### 구현 디테일과 최적화

**루프 순서의 함정**: 금액을 바깥 루프, 동전을 안쪽 루프로 뒤집으면 순열 수를 세게 된다. 예를 들어 `coins=[1,2], amount=3`에서 {1,2}와 {2,1}을 다른 경우로 취급해 4가지를 반환한다. 반드시 **동전 바깥, 금액 안쪽**이어야 한다.

**공간 절감**: 2D 테이블 $dp[k][x]$ 대신 1D $dp[x]$를 사용해 공간을 $O(amount)$로 줄인다. 이것이 가능한 이유는 동전 $c$를 처리할 때 $dp[x - c]$가 "같은 바깥 루프 내에서 이미 갱신된 값"을 참조하는 것이 정확히 원하는 동작(무한 사용)이기 때문이다.

**잠재적 오버플로**: 조합의 수가 매우 커질 수 있다. 문제에서 별도로 모듈러 연산을 요구하지 않더라도 JavaScript/TypeScript의 `number` 타입은 53비트 정수까지 정확하므로, 결과가 $2^{53}$을 초과할 가능성이 있는 경우 `BigInt`를 고려해야 한다.

## 수도 코드와 Activity Diagram

### 의사코드

```
function coinChangeWays(coins, amount):
    dp = 크기 (amount+1)인 배열, 0으로 초기화
    dp[0] = 1
    // 불변식: dp[x]는 "현재까지 처리한 동전들로 금액 x를 만드는 조합 수"

    for c in coins:                     // 바깥 루프: 동전 종류 (조합 보장)
        for x = c to amount:            // 안쪽 루프: 오름차순 (무한 사용 허용)
            dp[x] += dp[x - c]          // 전이: 동전 c 하나를 추가한 경우 누적

    return dp[amount]
```

### Activity Diagram

```mermaid
flowchart TD
    A([시작]) --> B["dp[0]=1, dp[1..amount]=0 초기화"]
    B --> C{동전 c 순회\n바깥 루프}
    C -- 모든 동전 완료 --> G([dp[amount] 반환])
    C -- 동전 c 선택 --> D{x = c..amount 순회\n안쪽 루프}
    D -- x 순회 완료 --> C
    D -- x 처리 중 --> E["dp[x] += dp[x - c]"]
    E --> D
```

**핵심 불변식**: 바깥 루프에서 동전 $c$의 처리를 완료한 시점에, $dp[x]$는 "코인 배열 내 $c$까지(포함)의 동전들만 사용해 금액 $x$를 만드는 조합의 수"이다.
