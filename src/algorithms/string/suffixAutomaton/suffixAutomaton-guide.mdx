# suffixAutomaton 해설

## 성능 목표 예측

| 제약 | 값 |
|------|----|
| 문자열 길이 $n = |s|$ | $1 \leq n \leq 10^5$ |
| 쿼리 문자열 길이 $|t|$ | 제한 없음 |

**naive 접근의 한계:**

`countDistinctSubstrings`: 모든 부분 문자열을 해시셋에 넣어 세는 방법.
- 부분 문자열 수 $O(n^2)$, 각 길이 $O(n)$ → 저장 $O(n^3)$ 시간, $O(n^2)$ 공간.

`contains(t)`: 텍스트에서 $t$를 KMP로 검색 → $O(n + |t|)$이지만, SAM 구축 없이는 전처리 없이 매 쿼리마다 $O(n)$.

낭비의 원인: 부분 문자열이 서로 다른 끝 위치 집합을 가지더라도 구조를 이용하지 않고 낱개로 처리한다. 같은 끝 위치 집합을 공유하는 부분 문자열들을 하나의 상태로 묶으면 $O(n)$ 공간이 가능하다.

**목표 복잡도:**

| 연산 | 시간 | 근거 |
|------|------|------|
| 구축 (`constructor`) | $O(n)$ | 문자당 amortized $O(1)$ |
| `countDistinctSubstrings` | $O(n)$ | 상태 수 $\leq 2n-1$ |
| `contains(t)` | $O(|t|)$ | 전이 추적만 |
| 공간 | $O(n)$ | 상태 수 $\leq 2n-1$, 전이 수 $\leq 3n-4$ |

## 목표 함수 (클래스)

```ts
class SuffixAutomaton {
  constructor(s: string)
  countDistinctSubstrings(): number
  contains(t: string): boolean
}
```

**파라미터 표:**

| 파라미터 | 의미 | 제약 |
|----------|------|------|
| `s` (constructor) | SAM을 구축할 문자열 | $1 \leq |s| \leq 10^5$ |
| `t` (contains) | 부분 문자열 여부를 검사할 문자열 | 제한 없음 |

**반환값:**
- `countDistinctSubstrings()`: $s$의 서로 다른 비공 부분 문자열 개수 $\#\{w \neq \varepsilon \mid w \text{ is a substring of } s\}$
- `contains(t)`: $t$가 $s$의 부분 문자열이면 `true`, 아니면 `false`

**엣지케이스:**

| 케이스 | `countDistinctSubstrings` | `contains` |
|--------|--------------------------|------------|
| `s = "a"` | 1 (`"a"`) | `contains("a")` → true, `contains("b")` → false |
| `s = "aa"` | 2 (`"a"`, `"aa"`) | `contains("aa")` → true |
| `s = "abcabc"` | 10 | `contains("abc")` → true |
| `contains("")` | — | `true` (빈 문자열은 항상 부분 문자열) |

## 핵심 아이디어

**핵심 아이디어**: "끝나는 위치 집합이 같은 부분 문자열끼리 하나의 상태로 묶으면, O(n) 공간에 모든 부분 문자열 정보를 담을 수 있다."

문자열의 부분 문자열을 낱개로 저장하면 O(n²) 공간이 필요하다. 접미사 자동화(SAM)는 "같은 위치들에서 끝나는 부분 문자열들"이 항상 서로의 접미사 관계임을 이용해 하나의 상태로 압축한다. 이 상태들의 수가 O(n)임이 증명되며, 한 글자씩 온라인으로 추가하는 구축 방법도 amortized O(1)이다.

**풀이 구조**
1. 초기 상태(루트)를 만들고 문자를 한 글자씩 추가한다.
2. 각 문자 추가 시 새 상태 cur을 생성하고 suffix link를 따라 올라가며 전이를 추가한다.
3. 전이가 이미 존재하는 상태 q를 만나면 길이 조건에 따라 clone을 생성하거나 직접 연결한다.
4. 서로 다른 부분 문자열 수는 각 상태 v에서 (len(v) - len(link(v)))를 합산한다.
5. 특정 문자열의 포함 여부는 루트에서 문자를 따라 전이하며 끝까지 성공하는지로 판단한다.

**조건**: 문자열의 모든 부분 문자열에 관한 쿼리(등장 횟수, 포함 여부, 서로 다른 부분 문자열 수 등)를 O(n) 공간에서 빠르게 처리해야 할 때.

**대표 예시**: `s = "abcabc"`에서 서로 다른 부분 문자열 수 = 10
SAM을 구축하면 상태 수가 10개 내외이다. 각 상태의 (len - link.len) 합이 10이 되어 `"a", "b", "c", "ab", "bc", "abc", "ca", "cab", "cabc", "abcabc"` 등 10개를 정확히 센다.

**언제 쓰나**
문자열 관련 쿼리 중 부분 문자열 존재 여부, 서로 다른 부분 문자열 수, 최장 공통 부분 문자열 등을 구해야 할 때 접미사 배열보다 구현이 간결하고 O(n) 공간·시간이 필요한 경우에 사용한다.

### 원형 아이디어와 naive 접근

$s$의 모든 부분 문자열을 하나씩 열거한다.

```
substrings = new Set()
for i = 0 to n-1:
  for j = i+1 to n:
    substrings.add(s[i..j-1])
return substrings.size
```

부분 문자열 수 $O(n^2)$, 해시 계산 $O(j-i)$ → 전체 $O(n^3)$ 시간, $O(n^2)$ 공간.
`contains(t)` 쿼리도 이 집합에서 단순 조회 가능하지만, 구축 비용이 크다.

### 어떤 관찰이 돌파구가 되는가

- **같은 "등장 위치 집합(endpos)"을 가진 부분 문자열들을 하나의 상태로 표현할 수 있다.** 예: `s = "abcbc"`에서 `"c"`와 `"bc"`는 모두 위치 {2, 4}에서 끝난다. 이 두 문자열을 하나의 SAM 상태로 묶으면, 그 상태가 나타내는 부분 문자열의 수는 최대 길이 - 최소 길이 + 1이다.
- **endpos가 다른 두 상태는 포함 관계이거나 서로소이다.** 이 성질 덕분에 SAM의 상태들이 suffix link tree를 형성하고, 상태 수가 $O(n)$으로 한정된다.
- **한 번에 문자를 추가하는 온라인 구축이 amortized $O(1)$이다.** 새 문자를 추가할 때 suffix link를 따라 올라가며 전이를 추가하는 과정이 amortized $O(1)$임이 포텐셜 방법으로 증명된다.

### 관찰을 형식화: 상태/구조 정의

**SAM 상태:**

```
State:
  len: number     // 이 상태가 나타내는 부분 문자열 중 최대 길이
  link: State     // suffix link: endpos 집합이 진상위집합인 상태 중 len이 가장 큰 것
  next: Map<char, State>  // 전이 함수
```

**endpos 동치류:**
두 부분 문자열 $u$, $v$ ($|u| \leq |v|$)가 같은 endpos를 가지면 $u$는 $v$의 접미사이다. 상태 $q$는 특정 endpos 집합을 공유하는 부분 문자열들의 집합을 나타내며, 그 길이 범위는 $[\text{len}(\text{link}(q))+1, \text{len}(q)]$이다.

이 정의가 이 형태여야 하는 이유: endpos 집합의 포함 관계가 suffix link를 통해 트리 구조로 표현되기 때문이다. 각 상태가 나타내는 부분 문자열의 수는 이 범위의 길이, 즉 $\text{len}(q) - \text{len}(\text{link}(q))$이다.

**`countDistinctSubstrings`의 형식화:**

$$\#\{w \neq \varepsilon \mid w \in \text{Sub}(s)\} = \sum_{q \neq q_0} \bigl(\text{len}(q) - \text{len}(\text{link}(q))\bigr)$$

여기서 $q_0$는 초기 상태 (len=0, 빈 문자열을 나타냄).

### 점화식 또는 핵심 연산

**온라인 구축 (문자 $ch$ 추가 시):**

새 상태 `cur` 생성 (len = last.len + 1).

suffix link를 따라 $p = \text{last}$에서 거슬러 올라가며 $p.\text{next}[ch]$가 없는 모든 상태에 $p.\text{next}[ch] = \text{cur}$ 추가. 이 과정에서 두 경우가 발생한다:

**경우 1:** $p = \text{null}$에 도달 (루트의 suffix link = null):
$$\text{link}(\text{cur}) = q_0$$

**경우 2:** $q = p.\text{next}[ch]$가 존재하는 $p$를 발견:
$$\text{link}(\text{cur}) = \begin{cases}
q & \text{if } q.\text{len} = p.\text{len} + 1 \\
\text{clone} & \text{otherwise}
\end{cases}$$

clone이 필요한 이유: $q.\text{len} > p.\text{len} + 1$이면 $q$의 endpos 집합에 새로운 위치를 추가해야 하는데, $q$는 이미 다른 부분 문자열들과 endpos를 공유하고 있어 분리해야 한다. clone은 $q$의 복사본으로 len만 $p.\text{len} + 1$로 줄이고, 이후 $p$에서 $q$로 향하던 전이를 clone으로 교체한다.

### 정당성 — 왜 이것이 옳은가

구축 후 SAM의 불변식:
1. $\text{len}(\text{link}(v)) < \text{len}(v)$ (모든 $v \neq q_0$)
2. root에서 전이를 따라가면 $s$의 모든 부분 문자열에 도달 가능
3. root에서 도달 가능한 모든 경로 라벨 = $s$의 모든 부분 문자열

**`countDistinctSubstrings` 정당성:** 각 상태 $v \neq q_0$는 길이 $[\text{len}(\text{link}(v))+1, \text{len}(v)]$의 부분 문자열을 나타내며, 이 집합들은 서로소이다. 따라서 합산이 중복 없이 전체 비공 부분 문자열 수를 준다.

**`contains` 정당성:** 초기 상태에서 $t$의 각 글자로 전이를 따라가면, SAM의 설계상 이 경로는 $t$가 $s$의 부분 문자열인 경우에만 끝까지 성공한다.

까다로운 케이스: clone 생성 없이 구현하면 어떤 부분 문자열이 두 상태에서 인식되거나 인식되지 않는 오류가 발생한다. clone 생성이 올바른 endpos 분리를 보장한다.

**amortized $O(n)$의 정당성:** 전이 추가 횟수는 각 문자 추가 시 while 루프에서 발생한다. 포텐셜 함수를 SAM의 전이 수 - 상태 수로 정의하면, 각 문자 추가 시 amortized $O(1)$ 연산임이 증명된다.

### 구현 디테일과 최적화

- **clone 생성 시 len만 변경:** clone은 $q$의 next와 link를 그대로 복사하되 len만 $p.\text{len} + 1$로 설정한다. $q$의 next를 복사하는 것이 중요하다(얕은 복사).
- **clone 이후 q.link = clone, cur.link = clone:** 두 가지를 모두 설정해야 한다. 하나라도 빠지면 suffix link tree가 손상된다.
- **`countDistinctSubstrings` 구현:** 모든 상태를 순회하며 `len - link.len`을 합산하면 된다. 루트(len=0)는 제외한다.
- **`contains("")`:** 빈 문자열은 루트에서 전이 없이 즉시 종료 → `true` 반환이 옳다. 루프를 한 번도 실행하지 않고 `return true`.
- **메모리 레이아웃:** SAM 상태를 배열로 관리하면 객체 참조 추적 대신 인덱스를 사용할 수 있어 캐시 친화적이다. $2n$ 크기의 배열을 미리 할당.

## 수도 코드와 Activity Diagram

### 의사코드

```
State:
  len: int = 0
  link: int = -1      // 인덱스, -1 = null
  next: Map<char, int>

// 전역 상태 배열
states: State[]
last: int             // 마지막으로 추가된 상태 인덱스

build(s):
  states = [new State()]   // states[0] = 루트, len=0, link=-1
  last = 0

  for ch in s:
    sa_extend(ch)

sa_extend(ch):
  cur = newState(len = states[last].len + 1)   // 새 상태 생성
  p = last

  // suffix link 따라 올라가며 전이 추가
  while p != -1 and ch not in states[p].next:
    states[p].next[ch] = cur
    p = states[p].link

  if p == -1:                                   // 루트 초과
    states[cur].link = 0                        // link = 루트
  else:
    q = states[p].next[ch]
    if states[q].len == states[p].len + 1:
      states[cur].link = q                      // 그대로 연결
    else:
      // clone: q 복사 후 len 조정
      clone = newState(
        len  = states[p].len + 1,
        link = states[q].link,
        next = copy of states[q].next           // 얕은 복사
      )
      while p != -1 and states[p].next[ch] == q:
        states[p].next[ch] = clone              // 전이 교체
        p = states[p].link
      states[q].link = clone
      states[cur].link = clone

  last = cur                                    // 불변식: last = 현재까지 입력된 접미사의 끝 상태

countDistinctSubstrings():
  total = 0
  for v = 1 to len(states)-1:                  // 루트(v=0) 제외
    total += states[v].len - states[states[v].link].len
  return total

contains(t):
  cur = 0                                       // 루트에서 시작
  for ch in t:
    if ch not in states[cur].next: return false
    cur = states[cur].next[ch]
  return true                                   // 전이를 모두 추적 성공
```

```mermaid
flowchart TD

    C -- Yes --> D["states[p].next[ch] = cur, p = states[p].link"]
    D --> C
```

**핵심 불변식:** 구축 후 항상 $\text{len}(\text{link}(v)) < \text{len}(v)$가 모든 $v \neq q_0$에 대해 성립하며, 루트에서 전이를 따라 도달 가능한 모든 경로 라벨의 집합이 $s$의 모든 부분 문자열과 일치한다.
