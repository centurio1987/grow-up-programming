# countInversions 해설

## 성능 목표 예측

| 항목 | 값 |
|------|-----|
| 입력 크기 $n$ | $1 \leq n \leq 10^5$ |
| 값 범위 | 정수, 제약 미명시 (비교 연산만 사용) |

**naive 접근의 한계.** 역순쌍을 이중 루프로 세면 모든 $(i, j)$ 쌍을 검사한다:

```
for i in 0..n-1:
    for j in i+1..n-1:
        if arr[i] > arr[j]: count++
```

시간복잡도 $O(n^2)$. $n = 10^5$이면 $10^{10}$ 번의 비교로 제한 시간 초과다.

**목표 복잡도와 근거.** Merge Sort는 재귀 깊이 $\log n$, 각 레벨의 병합 비용 $O(n)$으로 전체 $O(n \log n)$. 병합 과정에서 역순쌍 수를 동시에 누적하면 추가 비용 없이 같은 복잡도를 달성할 수 있다.

**공간 복잡도와 트레이드오프.** 병합 시 임시 배열 $O(n)$이 필요하다. In-place Merge Sort로 공간을 $O(1)$로 줄일 수 있지만, 구현 복잡도가 크게 증가하고 상수 계수도 나빠진다. $O(n)$ 추가 공간이 허용되면 표준 Merge Sort 변형을 선택한다.

---

## 목표 함수

```ts
function countInversions(arr: number[]): number
```

### 파라미터 표

| 파라미터 | 의미 | 제약 |
|---------|------|------|
| `arr` | 역순쌍을 셀 정수 배열 | 길이 $n$, $1 \leq n \leq 10^5$ |

**반환값.** $|\{(i, j) \mid 0 \leq i < j < n,\; arr[i] > arr[j]\}|$ — 인덱스 $i < j$이면서 값이 역순인 쌍의 총 개수.

### 엣지케이스

1. **`n = 0` 또는 `n = 1`** — 비교할 쌍이 없으므로 `0` 반환.
2. **이미 오름차순 정렬된 배열** — 역순쌍이 전혀 없으므로 `0` 반환.
3. **내림차순 정렬된 배열** — 모든 쌍이 역순이므로 $n(n-1)/2$ 반환. $n = 10^5$이면 약 $5 \times 10^9$로 `number`(부동소수점 53비트 정수 범위 $\approx 9 \times 10^{15}$) 안에 있어 안전하다.
4. **중복 원소가 있을 때** — `arr[i] === arr[j]`는 역순쌍이 아니다(`>` 조건). 병합 단계에서 좌측 원소와 우측 원소가 같을 때는 좌측을 먼저 복사해야 카운트를 잘못 늘리지 않는다.

## 핵심 아이디어

### 원형 아이디어와 naive 접근

"$i < j$인 모든 쌍에서 $arr[i] > arr[j]$를 직접 세자"는 생각이 출발점이다. 이 접근의 폭발 지점은 쌍의 개수가 $\binom{n}{2} = O(n^2)$이라는 점이다. 입력이 내림차순이면 이 숫자는 정확히 $n(n-1)/2$이며, $n = 10^5$에서 $\approx 5 \times 10^9$번의 비교를 강제한다.

### 어떤 관찰이 돌파구가 되는가

- **관찰 1.** 배열을 두 절반 $L = arr[0..\frac{n}{2}-1]$, $R = arr[\frac{n}{2}..n-1]$으로 나누면, 전체 역순쌍은 "L 내부", "R 내부", "L·R 교차" 세 그룹으로 분리된다. 세 그룹은 완전히 독립적이어서 합산 가능하다.
- **관찰 2.** L과 R이 각각 이미 정렬된 상태라면, 병합 과정에서 $L[i] > R[j]$가 발생하는 순간 $L[i], L[i+1], \ldots, L[|L|-1]$ 전부가 $R[j]$보다 크다는 사실을 $O(1)$에 알 수 있다. 정렬 상태가 이 일괄 계산을 가능하게 한다.
- **관찰 3.** Merge Sort가 정렬하면서 재귀 호출로 L·R 내부 역순쌍을 구하고, 병합 단계에서 교차 역순쌍을 구한다. 즉, 정렬이라는 "부산물"이 역순쌍 계산을 가속한다.

### 관찰을 형식화: 상태/구조 정의

재귀 함수 `mergeSort(arr, lo, hi)`는 두 가지 작업을 동시에 수행한다:

1. `arr[lo..hi]`를 제자리에서 오름차순 정렬한다.
2. `arr[lo..hi]` 안에 있는 역순쌍의 개수를 반환한다.

이 형태여야 하는 이유: 병합 단계에서 역순쌍을 세려면 L과 R이 정렬된 상태여야 한다. 따라서 정렬과 카운팅을 한 함수에서 함께 처리해야 하며, 재귀 호출 이후 그 상태(정렬 완료)를 보장받아야 한다.

### 점화식 또는 핵심 연산

역순쌍 수 $\text{inv}(arr)$의 유도:

$$\text{inv}(arr[lo..hi]) = \text{inv}(arr[lo..mid]) + \text{inv}(arr[mid+1..hi]) + \text{cross}(lo, mid, hi)$$

교차 역순쌍 $\text{cross}$의 계산. L과 R이 정렬된 상태에서 병합을 진행하며:

- 포인터 $i$ (L), $j$ (R)로 작은 값을 순서대로 선택한다.
- $L[i] \leq R[j]$이면 $L[i]$를 출력 — 교차 역순쌍 없음.
- $L[i] > R[j]$이면 $R[j]$를 출력 — $L[i], L[i+1], \ldots, L[|L|-1]$이 모두 $R[j]$보다 크므로 **$|L| - i$개의 교차 역순쌍** 발생.

$$\text{cross} \mathrel{+}= |L| - i \quad \text{(L[i] > R[j]인 매 순간)}$$

각 항의 의미: $|L| - i$는 L에서 아직 출력되지 않은 원소의 수이며, 이것들 모두 $R[j]$보다 크므로 $R[j]$와 각각 역순쌍을 이룬다. L이 정렬된 덕분에 개별 비교 없이 일괄 합산할 수 있다.

### 정당성 — 왜 이것이 옳은가

**귀납적 정당성.** $n = 1$에서는 역순쌍 0개이며 배열도 정렬 상태 — 기저 성립. 길이 $< n$의 배열에서 `mergeSort`가 올바르게 동작한다고 가정하면, 길이 $n$에서:

1. 재귀 호출로 $L$과 $R$이 각각 올바르게 정렬되고 내부 역순쌍이 정확히 계산된다(귀납 가정).
2. 병합 단계에서 정렬된 $L$과 $R$을 병합하므로 교차 역순쌍 계산이 올바르다(관찰 2 근거).

**까다로운 케이스.** 중복 원소가 있을 때 $L[i] = R[j]$ 분기에서 좌측 원소를 먼저 선택해야 한다. 만약 우측을 먼저 선택하면 본래 역순쌍이 아닌 경우(`arr[i] == arr[j]`, $i < j$)를 역순쌍으로 잘못 카운트하게 된다. `if L[i] <= R[j]` 조건(엄격 부등호 대신 `<=`)으로 처리한다.

### 구현 디테일과 최적화

- **임시 배열 재사용**: 매 `merge` 호출마다 임시 배열을 할당하면 가비지 컬렉션 부담이 크다. 전역 보조 배열을 미리 할당하고 재사용하면 실제 성능이 향상된다.
- **작은 구간에서의 삽입 정렬**: 구간 크기가 작을 때(예: $\leq 16$) 삽입 정렬로 전환하면 캐시 지역성이 좋아진다. 단, 역순쌍 카운팅 로직도 함께 처리해야 한다.
- **원본 배열 보존**: 문제 요구에 따라 정렬을 원본에 반영하면 안 되는 경우 배열 복사 후 처리한다.

## 수도 코드와 Activity Diagram

### 의사코드

```
function countInversions(arr):
    if |arr| <= 1:                    // 불변식: 길이 0·1은 역순쌍 0
        return 0
    return mergeSort(arr, 0, |arr| - 1)

function mergeSort(arr, lo, hi):
    // 불변식 진입: arr[lo..hi]는 아직 미정렬
    if lo >= hi:                      // 원소 1개 → 역순쌍 0, 이미 정렬
        return 0
    mid = (lo + hi) / 2
    count  = mergeSort(arr, lo, mid)       // 귀납: arr[lo..mid] 정렬 완료
    count += mergeSort(arr, mid+1, hi)     // 귀납: arr[mid+1..hi] 정렬 완료
    count += merge(arr, lo, mid, hi)       // 교차 역순쌍 계산 + 병합
    // 불변식 진출: arr[lo..hi]는 오름차순 정렬
    return count

function merge(arr, lo, mid, hi):
    L = 배열 arr[lo..mid]의 복사본      // 정렬된 좌측
    R = 배열 arr[mid+1..hi]의 복사본   // 정렬된 우측
    i = 0, j = 0
    k = lo                             // arr에 쓸 위치
    count = 0                          // 불변식: 지금까지 확정된 교차 역순쌍 수

    while i < |L| and j < |R|:
        if L[i] <= R[j]:               // 역순쌍 아님 (동치 포함)
            arr[k] = L[i]
            i++, k++
        else:                          // L[i] > R[j]: L[i..|L|-1] 전부 R[j]보다 큼
            count += |L| - i          // ← 핵심: 일괄 누적
            arr[k] = R[j]
            j++, k++

    // 남은 L 복사 (R[j..] 보다 모두 크지 않으므로 교차 역순쌍 추가 없음)
    while i < |L|: arr[k] = L[i], i++, k++
    // 남은 R 복사 (이미 제자리)
    while j < |R|: arr[k] = R[j], j++, k++

    return count
```

**핵심 불변식:** `merge` 호출 시 `arr[lo..mid]`와 `arr[mid+1..hi]`는 각각 오름차순 정렬 상태이며, `count`는 병합 과정에서 확정된 교차 역순쌍의 정확한 합계다.
