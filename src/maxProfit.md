## 현재 구현

```ts
export function maxProfit(A: number[]): number {
  if (A.length === 0) {
    return 0;
  }

  let maxDiff = 0;

  for (let l = A.length; l > 1; Math.floor(l / 2)) {
    console.log(l);
    for (let i = 0; i < A.length; i = i + l) {
      console.log(`i: ${i}`);
      const mid = Math.floor(l / 2) + i;
      let leftMin = 0;
      let rightMax = 0;
      for (let leftIdx = i; leftIdx < mid; leftIdx++) {
        if (leftMin > A[leftIdx]!) {
          leftMin = A[leftIdx]!;
        }
      }
      for (let rightIdx = mid; rightIdx < i + l; rightIdx++) {
        if (rightMax < A[rightIdx]!) {
          rightMax = A[rightIdx]!;
        }
      }
      console.log(leftMin, rightMax);
      maxDiff = Math.max(maxDiff, rightMax - leftMin);
    }
  }

  if (maxDiff > 0) {
    return maxDiff;
  }

  return 0;
}
```

## 1. 접근 방향성

**분할 정복(Divide & Conquer) 접근** — 윈도우 크기 $l$을 $N$부터 절반씩 줄여가며, 각 윈도우를 좌·우 반으로 나누어 좌측 최소값과 우측 최대값의 차이를 추적하는 전략이다.

**방향성은 부분적으로만 옳다.**

- 좌측 최솟값과 우측 최댓값의 차이를 본다는 아이디어 자체는 표준 $O(N)$ 풀이의 분할 정복 변형으로 성립한다.
- 그러나 윈도우 크기를 단순히 절반씩 줄여가며 **정렬된 청크(aligned chunks)** 만 검사하는 구조는 모든 $(P, Q)$ 쌍을 커버하지 못한다. 특정 $P, Q$가 매 레벨에서 같은 청크의 같은 반쪽에 속하면 그 쌍은 영영 비교되지 않는다.

  예: $A = [10, 1, 100, 2, 5]$, $l=5$: left=$[10,1]$ → leftMin=1, right=$[100,2,5]$ → rightMax=100 → diff=99. 정답이 우연히 나오지만, 청크 경계가 다른 위치에 있다면 누락 가능.

- 또한 $N$이 2의 거듭제곱이 아닐 때 마지막 청크가 $l$보다 짧아져 **배열 범위 밖 인덱스를 접근**한다 (`for (let rightIdx = mid; rightIdx < i + l; rightIdx++)`).

이 문제는 **최저점을 갱신하며 단일 패스**하는 $O(N)$ 풀이로 충분하다. 분할 정복은 과한 설계이며, 현재 구조에서는 정확성도 보장되지 않는다.

## 2. 정확성 분석

테스트는 **모두 타임아웃** 한다. 백그라운드 작업 로그에 `6` / `i: 0` / `0 21367` 가 무한 반복되어 출력 중이다. 원인은 다음 버그들이다.

### 버그 1: 외부 for 루프의 무한 루프 — **치명적**

```ts
for (let l = A.length; l > 1; Math.floor(l / 2)) {
```

`for` 업데이트식의 `Math.floor(l / 2)`은 **값을 계산만 하고 어디에도 대입하지 않는다.** `l`은 영원히 `A.length`에 머무르고, `l > 1`이 항상 참이므로 무한 루프에 빠진다.

올바른 형태:

```ts
for (let l = A.length; l > 1; l = Math.floor(l / 2)) {
```

테스트 출력에서 `6` (=N=문제 예시 길이)이 무한히 찍히는 것이 이 버그의 직접 증거다.

### 버그 2: `leftMin` 초기값 0 — 모든 케이스에서 오답

```ts
let leftMin = 0;
...
if (leftMin > A[leftIdx]!) {
  leftMin = A[leftIdx]!;
}
```

문제 제약 $0 \leq A[i] \leq 200{,}000$ 에서 **모든 원소가 비음수**이므로, `0 > A[leftIdx]` 조건은 절대 성립하지 않는다. → `leftMin`은 항상 0으로 고정.

결과적으로 `maxDiff = max(rightMax - 0) = rightMax`가 되어, 우측 절반의 최댓값을 그대로 반환한다.

테스트 출력의 `0 21367`이 정확히 이 버그를 보여준다. 좌측 절반의 실제 최솟값은 21011(=A[1]) 인데도 0이 출력된다. 만약 무한 루프 없이 종료된다면 반환값은 `21367 - 0 = 21367` (정답 356과 전혀 다름).

올바른 초기화:

```ts
let leftMin = A[i]!; // 또는 Infinity
let rightMax = A[mid]!; // 또는 -Infinity
```

### 버그 3: 마지막 청크에서 배열 범위 밖 접근

```ts
for (let i = 0; i < A.length; i = i + l) {
  const mid = Math.floor(l / 2) + i;
  ...
  for (let rightIdx = mid; rightIdx < i + l; rightIdx++) {
    ... A[rightIdx]! ...
  }
}
```

$N$이 $l$의 배수가 아닐 때, 마지막 청크는 $l$보다 짧다. 예를 들어 $N=5,\ l=2$ 라면 청크 시작은 $i=0,2,4$이고 마지막 청크 $[A[4], A[5]]$ 의 $A[5]$는 존재하지 않는다.

TypeScript의 non-null assertion(`!`)이 런타임 검사를 우회하므로 `undefined`가 산술 비교에 들어가고, 비교 결과는 NaN 전파로 계산이 망가진다.

올바른 처리:

```ts
const end = Math.min(i + l, A.length);
for (let rightIdx = mid; rightIdx < end; rightIdx++) { ... }
```

### 버그 4: 청크 경계로 인한 누락 가능성 (알고리즘 자체의 한계)

분할 정복 구조가 정렬된 청크만 검사하기 때문에, **윈도우 경계를 가로지르는 $(P, Q)$ 쌍**의 일부가 검사되지 않는다.

예: $A = [5, 1, 5, 1, 5]$, $l$ 시퀀스가 $5 \to 2 \to 1$ 로 진행될 때

- $l=5$: 청크 $[A[0..4]]$, mid=2. left=[5,1], right=[5,1,5]. leftMin=1, rightMax=5. diff=4 ✓
- $l=2$: 청크 $[A[0..1]],[A[2..3]],[A[4..]]$. 마지막 청크는 길이 1 → 범위 밖 접근
- $l=1$: 종료

이 예시는 $l=5$ 에서 정답을 잡지만, 일반적으로 **각 레벨에서 청크 시작 위치가 0의 배수에만 고정**되므로 $(P=1, Q=3)$ 같은 청크 경계를 가로지르는 쌍은 더 작은 윈도우에서 보지 못할 수 있다.

### 버그 5: `console.log` 로 인한 출력 오염 & 성능 저하

루프 안에서 매번 `console.log`가 호출되어 출력이 폭발하고 시간 제약도 위협한다. 디버그가 끝나면 제거해야 한다.

## 3. 최적화 분석

### 현재 복잡도

각 윈도우 크기 $l$ 단계에서 좌·우 합쳐 $O(l)$ 의 스캔을 수행하고, $l$ 크기의 청크가 $N/l$ 개 있으므로 한 레벨 비용은 $O(N)$. 윈도우 크기가 매번 절반이 되어 $O(\log N)$ 레벨 → 전체 $O(N \log N)$.

하지만:

- 무한 루프 버그로 실제로는 **종료 자체를 못한다**.
- 알고리즘 자체가 모든 쌍을 보장하지 못함 (위 버그 4).

### 목표 복잡도: $O(N)$ — 단순 단일 패스

이 문제는 **고전적인 best-time-to-buy-and-sell-stock** 으로, 좌→우 한 번의 패스로 풀린다.

#### 핵심 아이디어

각 $Q$ 시점에서, $0 \leq P \leq Q$ 인 모든 $P$ 중 **최저가**를 알면 그 $Q$ 에서의 최대 이익은 $A[Q] - \min_{0 \leq P \leq Q} A[P]$. 그래서 좌→우로 스캔하며 **지금까지의 최솟값** 만 유지하면 된다.

$$\text{maxProfit}(A) = \max_{0 \leq Q < N} \left( A[Q] - \min_{0 \leq P \leq Q} A[P] \right)$$

```ts
export function maxProfit(A: number[]): number {
  if (A.length < 2) return 0;

  let minSoFar = A[0]!;
  let best = 0;

  for (let i = 1; i < A.length; i++) {
    const price = A[i]!;
    if (price < minSoFar) {
      minSoFar = price;
    } else if (price - minSoFar > best) {
      best = price - minSoFar;
    }
  }

  return best;
}
```

#### 복잡도 비교

| 항목           | 현재 구현                              | 개선 후      |
| -------------- | -------------------------------------- | ------------ |
| 시간 복잡도    | $O(N \log N)$ (이론) / **무한** (실제) | $O(N)$       |
| 공간 복잡도    | $O(1)$                                 | $O(1)$       |
| N=400,000 성능 | 종료 못함                              | 수 ms        |
| 정확성         | 누락 가능 + leftMin 오류               | 모든 쌍 커버 |

## 5. 분할 정복 vs 단일 패스 — 본질적 비교

### 분할 정복으로 풀고 싶다면 (올바른 형태)

분할 정복도 정확히 구현하면 가능하다. 핵심은 **재귀 호출 시 항상 동일한 청크 분할이 아니라, 인덱스 범위 $[l, r]$ 을 절반으로 나누어 좌·우 부분 문제 + 좌측 min + 우측 max** 를 동시에 반환하는 것.

```ts
function solve(
  A: number[],
  l: number,
  r: number,
): { min: number; max: number; best: number } {
  if (l === r) return { min: A[l]!, max: A[l]!, best: 0 };
  const mid = (l + r) >> 1;
  const L = solve(A, l, mid);
  const R = solve(A, mid + 1, r);
  return {
    min: Math.min(L.min, R.min),
    max: Math.max(L.max, R.max),
    best: Math.max(L.best, R.best, R.max - L.min),
  };
}
```

이때 핵심은 `R.max - L.min` — 좌측의 어느 인덱스에서 사고 우측의 어느 인덱스에서 팔든 모든 쌍을 커버한다. 또한 좌측 내부 쌍은 `L.best`, 우측 내부 쌍은 `R.best`에서 본다.

### 그러나 이 문제에서는 $O(N)$ 이 더 우월

| 항목        | 분할 정복                  | 단일 패스 (권장) |
| ----------- | -------------------------- | ---------------- |
| 시간 복잡도 | $O(N)$ (T(N)=2T(N/2)+O(1)) | $O(N)$           |
| 공간 복잡도 | $O(\log N)$ (재귀 스택)    | $O(1)$           |
| 구현 복잡도 | 중간                       | 매우 간단        |
| 가독성      | 보통                       | 매우 직관적      |

본 문제는 "지금까지의 최솟값" 이라는 단순 누적 상태로 풀리는 전형적 단일 패스 문제이므로, 분할 정복은 과한 도구다.

## 반복 2 — `l /= 2` 로 무한 루프 해제 (9 pass / 14 fail)

> 분석 일자: 2026-05-14
> 테스트 결과: 9 pass / 14 fail (성능 테스트 3868ms 소요)

### 변경된 구현

```ts
for (let l = A.length; l > 1; l /= 2) {  // ← 업데이트식 수정
  ...
  for (let i = 0; i < A.length; i += l) {
    ...
    const mid = Math.floor(l / 2) + i;
    let leftMin = 0;
    let rightMax = 0;
    for (let leftIdx = i; leftIdx < mid; leftIdx++) { ... }
    for (let rightIdx = mid; rightIdx < i + l; rightIdx++) { ... }
    ...
  }
}
```

### 개선된 점

| 버그              | 반복 1                            | 반복 2                           |
| ----------------- | --------------------------------- | -------------------------------- |
| 무한 루프         | `Math.floor(l/2)` 대입 누락 ❌    | `l /= 2` 로 변경 ✅ (종료는 됨) |

`l` 이 매 단계 절반이 되면서 `l > 1` 에서 결국 종료된다. 더 이상 테스트가 타임아웃하지 않는다.

### 남은 / 새로운 버그

**버그 A (잔존): `leftMin = 0` 초기화** — 가장 영향이 크다.

```ts
let leftMin = 0;
if (leftMin > A[leftIdx]!) leftMin = A[leftIdx]!;
```

입력 제약 $A[i] \geq 0$ 에서 갱신 조건이 절대 성립하지 않아 `leftMin` 은 항상 0. 결과적으로 `diff = rightMax - 0 = rightMax` 가 되고, 함수는 **배열에서 본 적 있는 최댓값을 그대로 반환**한다.

테스트 결과가 이를 정확히 보여준다:

| 입력                                       | 기대값 | 실제 반환     | 이유                  |
| ------------------------------------------ | ------ | ------------- | --------------------- |
| `[23171,21011,21123,21366,21013,21367]`    | 356    | **23171**     | 좌측 절반의 max = 23171 |
| `[1,2,3,4,5]`                              | 4      | **5**         | rightMax = 5          |
| `[5,4,3,2,1]`                              | 0      | **5**         | leftMin=0, rightMax=5 |
| `[7,7,7,7]`                                | 0      | **7**         | rightMax=7            |
| `[7,1,5,3,6,4]`                            | 5      | **7**         | rightMax=7            |
| `[10,1]`                                   | 0      | **1**         | rightMax=1            |
| `[5,4,3,2,1,0]`                            | 0      | **5**         | rightMax=5            |
| `[1,100,2,3]`                              | 99     | **100**       | rightMax=100          |
| `[5,6,1,100]`                              | 99     | **100**       | rightMax=100          |
| `[200000,200000,200000]`                   | 0      | **200000**    | rightMax=200000       |

수정:

```ts
let leftMin = A[i]!;     // 또는 Infinity
let rightMax = A[mid]!;  // 또는 -Infinity
```

**버그 B (신규): `l` 이 정수가 아니다 — 인덱스가 분수가 된다**

`l /= 2` 는 `Math.floor` 없이 부동소수점 나눗셈을 수행한다. $N=6$ 에서 시퀀스: $6 \to 3 \to 1.5 \to$ (종료, 0.75 < 1).

`l = 1.5` 단계에서 `i += l` 이 분수를 누적해 `i` 가 0, 1.5, 3, 4.5 로 전개된다. 테스트 로그의 다음 줄이 직접 증거다:

```
1.5
i: 0
0 23171
i: 1.5      ← 분수 인덱스
0 0
i: 3
0 21366
i: 4.5      ← 분수 인덱스
0 0
```

`leftIdx`/`rightIdx` 가 정수 시작값이라도, `i` 가 분수이면 `i + l` 같은 종료 조건이 비대칭이 되고, `A[1.5]` 같은 비정수 접근(JS에서는 `undefined`)이 발생한다. `!` 단언이 이를 가리는 동안 산술은 NaN으로 오염된다.

`for (let l = A.length; l > 1; l = Math.floor(l / 2))` 로 정수 시퀀스 $6 \to 3 \to 1$ 로 만들어야 한다.

**버그 C (잔존): 마지막 청크 범위 밖 접근**

여전히 `for (let rightIdx = mid; rightIdx < i + l; rightIdx++)` 가 `A.length` 를 넘는다. 청크 길이 미만 처리 누락.

**버그 D (잔존): 알고리즘 자체의 누락 가능성**

정렬된 청크 구조라 청크 경계를 가로지르는 일부 $(P, Q)$ 쌍이 영영 비교되지 않는다 (반복 1의 버그 4 참조).

**버그 E (잔존): `console.log` 폭증 — 성능 직접 영향**

성능 테스트 케이스에서 `l = 1.5...` 단계까지 진행되며 `i` 가 분수로 갱신되어 `400,000 / 1.5...` 회 정도의 i 반복마다 `console.log` 가 호출된다. 35MB 출력이 누적되어 테스트가 **3,868ms** 소요됐다. console.log를 제거해도 알고리즘 시간복잡도 자체는 $O(N \log N)$ 이라 100ms 통과가 가능하나, 현재 출력 오염만으로도 성능 테스트 실패.

테스트 로그 말미의 `i: 399998.47412109375` 처럼 인덱스가 400,000 직전까지 0.001 단위로 누적되는 모습은 부동소수점 누산 오차까지 더해진 결과다.

### 권장 조치 (우선순위 순)

1. **즉시:** `let leftMin = A[i]!;` `let rightMax = A[mid]!;` 로 초기화 수정 → 약 8~10개 테스트가 추가로 통과할 것으로 예상.
2. **즉시:** `l = Math.floor(l / 2)` 로 정수 시퀀스 보장.
3. **즉시:** `console.log` 전부 삭제.
4. **이후:** 마지막 청크 클램프 (`Math.min(i + l, A.length)`).
5. **재검토 필요:** 청크 경계 누락 문제 — 단순 단일 패스($O(N)$)로의 전환을 권장.

## 반복 3 — `leftMin = MAX_SAFE_INTEGER` 초기화 (19 pass / 4 fail)

> 분석 일자: 2026-05-14
> 테스트 결과: 19 pass / 4 fail

### 변경된 구현

```ts
for (let l = A.length; l > 1; l /= 2) {
  console.log(l);
  for (let i = 0; i < A.length; i += l) {
    console.log(`i: ${i}`);
    const mid = Math.floor(l / 2) + i;
    let leftMin = Number.MAX_SAFE_INTEGER;  // ← 0 에서 MAX_SAFE_INTEGER 로 변경
    let rightMax = 0;
    ...
  }
}
```

### 개선된 점

| 버그                | 반복 2                                     | 반복 3                            |
| ------------------- | ------------------------------------------ | --------------------------------- |
| `leftMin` 항상 0    | `let leftMin = 0` → 갱신 안 됨 ❌          | `Number.MAX_SAFE_INTEGER` ✅      |

비음수 입력에서 `MAX_SAFE_INTEGER > A[leftIdx]` 가 항상 성립하므로 좌측 최솟값이 정상적으로 갱신된다. **반복 2의 14 fail → 반복 3의 4 fail** 로 10개 테스트가 새로 통과한다 (문제 예시, 단조 증가/감소, [7,1,5,3,6,4], [10,1], [1,100,2,3], [5,6,1,100] 등).

테스트 로그 발췌 (반복 2와의 비교):

```
반복 2: i: 0  →  "0 21367"  (leftMin=0, rightMax=21367)
반복 3: i: 0  →  "21011 21367"  (leftMin=21011, rightMax=21367) → diff=356 ✅
```

### 남은 / 새로운 버그

**버그 A (잔존, 신규 노출): `l` 이 분수 — 인덱스가 분수가 된다**

`l /= 2` 가 그대로 남아 있어 $l$이 $6 \to 3 \to 1.5 \to 0.75$ (종료). 분수 `l` 단계에서:

```
1.5
i: 0
9007199254740991 23171    ← mid=floor(1.5/2)+0=0 → 좌측 0..0 빈 범위, leftMin 초기값 그대로
i: 1.5
9007199254740991 0        ← A[1.5]=undefined → rightMax 갱신 안 됨, diff=−MAX
i: 3
9007199254740991 21366
i: 4.5
9007199254740991 0
```

이 단계에서는 `mid = Math.floor(l/2) + i` 가 `floor(0.75) + i = i`. 좌측 루프 `leftIdx < mid` 가 `leftIdx < i` 라 절대 실행되지 않아 `leftMin` 이 `MAX_SAFE_INTEGER` 그대로다. 우측은 `A[1.5]`, `A[4.5]` 같은 undefined를 만나 갱신 실패. `diff = 0 - MAX_SAFE_INTEGER` 은 음수이므로 `maxDiff` 에 영향 없음 → **현재까지는 정답을 망가뜨리지 않지만, 성능을 망친다** (아래 버그 D 참조).

수정: `l = Math.floor(l / 2)` 로 변경하면 시퀀스가 $6 \to 3 \to 1$ (종료) 로 정수가 된다.

**버그 B (신규 표면화): 알고리즘이 "P, Q 가 둘 다 우측 절반"인 경우를 누락 — 4개 fail 중 2개의 원인**

분할 정복 구조가 매 청크에서 "좌측 min × 우측 max" 만 본다. 그러나 최적 $(P, Q)$ 가 **같은 우측 절반** 또는 **같은 좌측 절반** 안에 있을 수 있고, 이를 잡으려면 하위 청크에서 다시 봐야 한다. 하지만 현재 구현은 하위 청크가 분수 `l` 단계로 망가져서 그것조차 못 잡는다.

**Case 1: `[200000, 0, 200000]` → 기대 200000, 실제 0**

```
l=3, mid=1: left=[200000], right=[0,200000]. leftMin=200000, rightMax=200000. diff=0
l=1.5 단계는 분수 인덱스로 의미 없는 출력만 생성
maxDiff = 0 → 반환 0 ❌
```

정답 $(P=1, Q=2)$ 는 둘 다 "오른쪽 절반" 안에 있어 top-level 분할에서 잡을 수 없다. 하위 청크 `right=[0, 200000]` 를 재귀로 풀어야 잡힌다.

**Case 2: V자 패턴 `[10,5,1,5,10]` → 기대 9, 실제 5**

```
l=5, mid=2: left=[10,5], right=[1,5,10]. leftMin=5, rightMax=10. diff=5
l=2.5: mid=1. left=[10], right=[5]. diff=-5.
       i=2.5 부터 분수 인덱스로 망가짐.
maxDiff = 5 → 반환 5 ❌
```

정답 $(P=2, Q=4)$, 즉 1 사고 10 팔기 — 둘 다 우측 절반 `[1,5,10]` 안에 있어 top-level 분할에서 못 잡는다.

수정: 분할 정복을 제대로 한다면 [반복 1의 5장 "올바른 형태"](#5-분할-정복-vs-단일-패스--본질적-비교) 처럼 **인덱스 범위 $[l,r]$ 재귀**로 좌·우 부분 문제와 cross 케이스를 모두 본다. 또는 $O(N)$ 단일 패스로 전환.

**버그 C (잔존): 마지막 청크 범위 밖 접근**

`for (let rightIdx = mid; rightIdx < i + l; rightIdx++)` 가 그대로다. 정수 `l` 로 바꾸어도 $N$ 이 $l$ 의 배수가 아니면 범위 밖 접근이 발생한다. `Math.min(i + l, A.length)` 로 클램프 필요.

**버그 D (성능): 분수 `l` + `console.log` 폭증 → 성능 테스트 3800ms+**

| 항목                  | 기대값  | 실제 측정값      |
| --------------------- | ------- | ---------------- |
| N=400,000 톱니파       | <100ms  | **4190.89ms**    |
| N=400,000 단조 감소    | <100ms  | **3815.18ms**    |

원인 두 가지가 합쳐진다:

1. **분수 `l` 누적**: `l = 400000` 부터 시작해 `/= 2` 를 반복하면 $l$ 은 0.5 미만이 될 때까지 약 20 단계 진행. 각 단계에서 `i += l` 가 분수 누적 → 부동소수점 오차로 인해 마지막 단계에서 $i$가 `399998.47412...` 같은 형태로 0.001 단위까지 증가. 한 단계에서만 800,000번 가까운 i 반복.
2. **`console.log` 호출 폭증**: 각 i 반복마다 3개 로그. 출력 버퍼링과 stdout I/O 가 직접 시간을 잡아먹는다. 테스트 로그가 **58MB** 에 달한 것이 직접 증거.

수정 우선순위: `l = Math.floor(l / 2)` 로 단계 수를 $\log_2 N \approx 19$ 로 제한 + `console.log` 전부 제거. 이 둘만으로도 성능은 $O(N \log N) \approx 7.6\text{M ops}$ → 수십 ms 안에 들어온다.

### 권장 조치 (우선순위 순)

| # | 조치                                          | 효과                                |
| - | --------------------------------------------- | ----------------------------------- |
| 1 | `l = Math.floor(l / 2)` 로 정수 시퀀스 보장   | 성능 + 인덱스 위생                  |
| 2 | `console.log` 3개 제거                        | 성능 (단독으로도 큰 효과)            |
| 3 | 알고리즘을 인덱스 범위 재귀로 재작성, 또는 $O(N)$ 단일 패스 | Case 1·2 정답 누락 해결              |
| 4 | 마지막 청크 클램프 `Math.min(i+l, A.length)`  | 안전성                              |

1·2번만 해도 성능 테스트 2개는 통과 가능. 3번까지 마치면 모든 23 테스트 통과 예상.

## 반복 4 — $O(N)$ 단일 패스로 컨셉 전환 (23 pass / 0 fail)

> 분석 일자: 2026-05-15
> 테스트 결과: **23 pass / 0 fail** (전체 102ms)

### 변경된 구현

분할 정복 구조를 완전히 폐기하고, 최저점 누적 + 최대 이익 누적의 **단일 패스** 로 재작성했다.

```ts
export function maxProfit(A: number[]): number {
  let minSoFar = A[0]!;
  let maxProfit = 0;

  for (let i = 1; i < A.length; i++) {
    const price = A[i]!;
    if (price < minSoFar) {
      minSoFar = price;
    } else if (price - minSoFar > maxProfit) {
      maxProfit = price - minSoFar;
    }
  }

  return maxProfit;
}
```

(기존 분할 정복 코드는 주석 처리되어 남아 있다.)

### 1. 접근 방향성

**완전히 올바르다.** 본 문제는 "지금까지의 최솟값 vs 현재 가격" 이라는 단순 상태로 풀리는 전형적 단일 패스 문제다. 수학적으로:

$$\text{maxProfit}(A) = \max_{0 \leq Q < N} \left( A[Q] - \min_{0 \leq P \leq Q} A[P] \right)_+$$

여기서 $(\cdot)_+ = \max(\cdot, 0)$. 좌→우로 스캔하며 두 상태만 유지하면 모든 $(P, Q)$ 쌍을 암묵적으로 본 것과 동치다. 반복 1~3의 분할 정복 청크 정렬 구조에서 누락되던 **"P, Q 가 둘 다 우측 절반"** 같은 케이스가 자연스럽게 해결된다.

### 2. 정확성 분석

23/23 모두 통과. 주요 케이스 추적:

| 입력                                    | 정답   | 추적                                                |
| --------------------------------------- | ------ | --------------------------------------------------- |
| `[23171,21011,21123,21366,21013,21367]` | 356    | i=1에서 min=21011, i=5에서 21367-21011=356 ✓        |
| `[200000, 0, 200000]`                   | 200000 | i=1에서 min=0, i=2에서 200000-0=200000 ✓             |
| V자 `[10,5,1,5,10]`                     | 9      | i=1,2에서 min 갱신(5→1), i=4에서 10-1=9 ✓            |
| 단조 감소 `[5,4,3,2,1]`                 | 0      | 매번 min만 갱신, profit 분기 안 들어감 → 0 ✓        |
| 빈 배열 `[]`                            | 0      | for 루프 미진입(i=1 < 0 거짓), `maxProfit=0` 반환 ✓ |
| `N=1, [42]`                             | 0      | 동일 ✓                                              |

**잠재 문제 한 가지 (현재는 영향 없음):**

```ts
let minSoFar = A[0]!;
```

`A.length === 0` 일 때 `A[0]` 은 `undefined` 이며, `!` 단언이 이를 가린다. 런타임에서 `minSoFar = undefined` 가 되지만, 곧바로 `for (let i = 1; i < 0; ...)` 가 거짓이라 루프가 실행되지 않고 `maxProfit=0` 이 반환되어 결과는 올바르다. TS 엄격 모드에서 보면 안전하지 않은 단언이지만, 현 동작에는 문제 없음. 명시성을 원한다면 `if (A.length === 0) return 0;` 가드를 두면 된다.

### 3. 최적화 분석

| 항목         | 분할 정복(반복 3)                     | 단일 패스(반복 4)         |
| ------------ | ------------------------------------- | -------------------------- |
| 시간 복잡도  | $O(N \log N)$ (이론) / 무한·분수(실제)| **$O(N)$**                |
| 공간 복잡도  | $O(1)$ (제자리)                       | **$O(1)$**                |
| N=400,000 성능 | 4190ms (실패)                       | **전체 23 테스트 102ms**  |
| 알고리즘 누락 | 청크 경계 문제                       | 없음 (모든 쌍 암묵 비교)   |
| 구현 복잡도  | 높음                                  | **매우 낮음** (~10 LOC)   |

분기 구조 `if (price < minSoFar) ... else if (...)` 는 두 갱신을 한 번에 묶어 분기 예측에도 우호적이다. N=400,000 두 케이스를 합쳐도 충분히 100ms 이내.

### 4. 마무리 코멘트

- 코드 자체는 거의 최적. 더 손댈 곳은 없다.
- 정리 작업으로 권장:
  - 주석 처리된 이전 분할 정복 코드는 제거 (혹은 학습용으로 보존할 거면 별도 파일로 분리).
  - `A[0]!` 단언 대신 `if (A.length === 0) return 0;` 명시 가드를 두면 의도가 더 분명해진다 (선택).

이 단일 패스가 본 문제의 표준 풀이이며, 반복 3까지 노출된 모든 알고리즘 누락 / 성능 문제가 컨셉 전환 한 번으로 해소되었다.
