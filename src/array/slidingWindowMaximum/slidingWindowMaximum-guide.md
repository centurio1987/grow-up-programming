# slidingWindowMaximum — 슬라이딩 윈도우 최댓값

## 성능 목표 예측

| 항목 | 값 |
|------|-----|
| 입력 크기 | $1 \leq N \leq 100{,}000$ |
| 윈도우 크기 | $1 \leq k \leq N$ |
| 원소 범위 | $-10{,}000 \leq nums[i] \leq 10{,}000$ |

**naive 접근의 문제점**: 각 윈도우 위치마다 $k$개의 원소 중 최댓값을 선형 탐색하면 $O(Nk)$이다. $k = N/2$이면 $O(N^2/2) = 5 \times 10^9$으로 시간 초과가 발생한다.

**목표 복잡도**: 시간 $O(N)$, 공간 $O(k)$. 각 원소가 덱에 최대 한 번 삽입되고 한 번 제거되므로 전체 $O(N)$이다.

**공간 복잡도**: 덱에 최대 $k$개의 인덱스가 저장되므로 $O(k)$이다.

---

## 목표 함수

```ts
function slidingWindowMaximum(nums: number[], k: number): number[]
```

| 파라미터 | 의미 | 제약 |
|----------|------|------|
| `nums` | 정수 배열 | $1 \leq N \leq 100{,}000$, $-10{,}000 \leq nums[i] \leq 10{,}000$ |
| `k` | 윈도우 크기 | $1 \leq k \leq N$ |

**반환값**: 윈도우가 왼쪽부터 오른쪽으로 한 칸씩 이동할 때 각 위치의 최댓값 배열. 길이 $N - k + 1$.

**엣지케이스**:

| 입력 | 기대 출력 | 이유 |
|------|-----------|------|
| `k=1` | `nums` 그대로 | 윈도우 크기 1 |
| `k=N` | `[max(nums)]` | 전체 배열이 하나의 윈도우 |
| `[1,1,1,1]`, `k=2` | `[1,1,1]` | 모두 같은 값 |
| `[5,4,3,2,1]`, `k=3` | `[5,4,3]` | 단조 감소 |

---

## 핵심 아이디어

**핵심 아이디어**: "현재 원소보다 작으면서 더 왼쪽에 있는 원소는 앞으로 어떤 윈도우에서도 최댓값이 될 수 없으므로 즉시 제거한다."

슬라이딩 윈도우 최댓값을 매번 재계산하면 O(Nk)가 된다. 단조 감소 덱을 유지하면, 새 원소가 들어올 때 덱 뒤에서 자신보다 작거나 같은 원소를 모두 제거하고 추가한다. 덱 앞이 항상 현재 윈도우의 최댓값 인덱스를 가리키므로 O(1)에 최댓값을 읽을 수 있다.

**풀이 구조**
1. 빈 덱과 빈 결과 배열로 초기화한다.
2. i=0..N-1을 순회하며: 덱 앞에서 윈도우 범위(i-k+1) 밖의 인덱스를 제거한다.
3. 덱 뒤에서 nums[덱.back()] <= nums[i]인 인덱스를 제거한다(단조 감소 유지).
4. 현재 인덱스 i를 덱 뒤에 추가한다.
5. i >= k-1이면 nums[덱.front()]를 결과에 기록한다.

**조건**: 크기 k의 고정 윈도우가 왼쪽에서 오른쪽으로 한 칸씩 이동. 결과 배열 길이는 N-k+1.

**대표 예시**: `nums=[1,3,-1,-3,5,3,6,7], k=3`
윈도우 [1,3,-1]의 최대 3, [-3]추가 후 [3,-1,-3]의 최대 3, ..., [5,3,6]의 최대 6, [3,6,7]의 최대 7. 결과 [3,3,5,5,6,7].

**언제 쓰나**
고정 크기 윈도우가 배열을 슬라이딩하면서 구간의 최댓값(또는 최솟값)을 O(N)에 모두 구해야 하는 문제에서 사용한다. 윈도우 크기가 가변이라면 투 포인터 + 덱 조합으로 확장할 수 있다.

### 원형 아이디어와 naive 접근

각 윈도우 시작 위치 $i - k + 1$에 대해 $nums[i-k+1..i]$ 중 최댓값을 직접 계산한다.

```
result = []
for i from k-1 to N-1:
    maxVal = nums[i]
    for j from i-k+1 to i-1:
        maxVal = max(maxVal, nums[j])
    result.push(maxVal)
```

각 윈도우에서 $k$번 비교하므로 전체 $O(Nk)$이다. 인접한 두 윈도우 사이에 $k-1$개의 원소가 겹치는데, 이 겹치는 부분의 최댓값을 매번 재계산하는 것이 낭비다.

### 어떤 관찰이 돌파구가 되는가

- **관찰 1**: 윈도우가 오른쪽으로 한 칸 이동할 때, 왼쪽 원소 하나가 제거되고 오른쪽 원소 하나가 추가된다. 최댓값 후보를 효율적으로 관리할 수 있다면 $O(1)$에 윈도우 최댓값을 얻을 수 있다.
- **관찰 2**: 현재 윈도우 안에서, 인덱스 $j < i$이고 $nums[j] \leq nums[i]$이면 $j$는 앞으로 어떤 윈도우에서도 최댓값이 될 수 없다 ($i$가 더 오른쪽에 있어 먼저 윈도우를 벗어나지 않으므로). 따라서 $j$를 후보에서 제거해도 안전하다.
- **관찰 3**: 이 원칙을 적용하면 후보 인덱스들의 `nums` 값이 단조 감소인 덱(deque)이 자연스럽게 만들어진다. 덱의 앞이 항상 현재 윈도우의 최댓값이다.

### 관찰을 형식화: 상태/구조 정의

단조 감소 덱(인덱스 저장)을 유지한다.

$$\text{덱 내 인덱스} \; d_0 < d_1 < \ldots < d_m \; \text{에 대해} \; nums[d_0] \geq nums[d_1] \geq \ldots \geq nums[d_m]$$

**불변식**:
1. 덱 내 모든 인덱스는 현재 윈도우 $[i-k+1, i]$ 안에 있다.
2. 덱 내 `nums` 값은 앞에서 뒤로 단조 비증가다.
3. 덱이 비어있지 않으면 `nums[deque.front()]`가 현재 윈도우 최댓값이다.

이 정의가 왜 이 형태여야 하는가: 앞에서 최댓값을 $O(1)$에 읽으려면 최댓값 후보가 앞쪽에 있어야 한다. 단조 감소를 유지하면 앞이 항상 가장 큰 값이다. 단조 증가 덱을 쓰면 앞이 최솟값이 되어 최댓값 접근에 $O(k)$가 필요하다.

### 점화식 또는 핵심 연산

$i = 0, 1, \ldots, N-1$에 대해:

```
1. 덱 앞에서 윈도우 벗어난 인덱스 제거:
   while deque.front() < i - k + 1: deque.popFront()

2. 덱 뒤에서 현재 원소보다 작거나 같은 인덱스 제거:
   while deque.back() exists and nums[deque.back()] <= nums[i]: deque.popBack()

3. 현재 인덱스 추가:
   deque.pushBack(i)

4. 윈도우가 완성되면 결과 기록:
   if i >= k - 1: result.push(nums[deque.front()])
```

- 단계 1: 윈도우 불변식 유지 — 오래된 인덱스 제거
- 단계 2: 단조 감소 불변식 유지 — 새 원소보다 작은 후보는 앞으로 최댓값이 될 수 없음
- 단계 3: 현재 인덱스를 후보로 등록
- 단계 4: 윈도우가 $k$개 원소로 채워진 시점($i \geq k-1$)부터 결과를 기록

### 정당성 — 왜 이것이 옳은가

덱의 앞이 현재 윈도우 최댓값임을 귀납적으로 증명한다.

단계 2에서 제거된 인덱스 $j$는 $nums[j] \leq nums[i]$이다. $j < i$이고 `nums[j] <= nums[i]`이면, $j$가 윈도우 안에 있는 동안 $i$도 윈도우 안에 있다 ($j < i$이므로 $j$가 윈도우를 벗어나기 전에 $i$는 항상 더 크거나 같은 값을 제공한다). 따라서 $j$를 제거해도 최댓값을 놓치지 않는다.

단계 1에서 제거된 인덱스들은 윈도우를 벗어났으므로 더 이상 고려할 필요 없다.

결국 덱에 남은 인덱스들은 현재 윈도우 안에서 단조 감소를 유지하고, 앞이 최댓값이다.

같은 값 처리: 단계 2에서 `<=` (같은 경우도 제거)를 사용하면 새 인덱스가 우선시된다. `<`로 바꾸면 오래된 인덱스가 우선되지만 결과는 동일하다.

### 구현 디테일과 최적화

- **덱에 값이 아닌 인덱스를 저장**: 단계 1의 윈도우 범위 확인 (`i - k + 1`)이 인덱스를 요구한다. 값을 저장하면 인덱스를 역추적할 수 없다.
- **함정**: 단계 1(앞 제거)과 단계 2(뒤 제거) 순서를 바꾸면 안 된다. 단계 2를 먼저 하면 이미 윈도우를 벗어난 인덱스와 비교하는 오류가 발생할 수 있다.
- **함정**: `nums[deque.back()] <= nums[i]`에서 `<` 대신 `<=`를 써야 중복 값에서 덱이 불필요하게 길어지는 것을 방지한다 (선택 사항이나 권장).
- **함정**: `i >= k - 1` 조건 없이 결과를 기록하면 윈도우가 $k$개로 채워지지 않은 상태에서 기록해 길이가 달라진다.
- $k = 1$인 경우: 덱에 항상 원소가 하나뿐이고 매 스텝 결과를 기록하므로 `nums`를 그대로 반환하는 것과 동일하다.

## 수도 코드와 Activity Diagram

### 의사코드

```
function slidingWindowMaximum(nums, k):
    deque  ← 빈 양방향 큐 (인덱스 저장)    // 불변식: 앞→뒤 nums값 단조 비증가, 모두 현재 윈도우 내
    result ← []

    for i from 0 to N-1:
        // 윈도우 밖의 앞쪽 인덱스 제거
        while deque 비어있지 않고 deque.front() < i - k + 1:
            deque.popFront()

        // 새 원소보다 작거나 같은 뒤쪽 인덱스 제거 (단조 감소 유지)
        while deque 비어있지 않고 nums[deque.back()] <= nums[i]:
            deque.popBack()

        deque.pushBack(i)

        if i >= k - 1:                         // 윈도우가 완성된 시점부터
            result.push(nums[deque.front()])    // 덱 앞 = 현재 윈도우 최댓값

    return result
```
