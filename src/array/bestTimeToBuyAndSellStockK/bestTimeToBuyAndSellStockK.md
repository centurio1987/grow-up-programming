# bestTimeToBuyAndSellStockK 분석 보고서

## 1. 접근 방법의 방향성

문제는 최대 $k$회의 (매수, 매도) 짝을 골라 이익의 합

$$
\sum_{m=1}^{k'} (prices[j_m] - prices[i_m]),\quad i_1 \leq j_1 < i_2 \leq j_2 < \cdots,\ k' \leq k
$$

을 최대화하는 것이다. 가이드(`bestTimeToBuyAndSellStockK-guide.md`)는 상태 $(t,\ \text{보유 여부})$를 갖는 DP를 권장한다.

- $buy[t][i]$ : $i$일까지 $t$번째 매수를 마친 상태의 최대 이익
- $sell[t][i]$ : $i$일까지 $t$번째 매도를 마친 상태의 최대 이익

점화식:

$$
buy[t][i] = \max(buy[t][i-1],\ sell[t-1][i-1] - prices[i])
$$
$$
sell[t][i] = \max(sell[t][i-1],\ buy[t][i-1] + prices[i])
$$

시간복잡도 $O(Nk)$, 공간복잡도 $O(k)$(롤링 배열)로 입력 한도 $N \leq 1{,}000,\ k \leq 100$에 충분하다.

**현재 솔루션의 방향성은 가이드와 일치하지 않는다.** 코드(`bestTimeToBuyAndSellStockK.ts:37-67`)는 "가장 큰 단일 거래 이익을 찾고 그 매수/매도 지점을 배열에서 제거한 뒤 $k$번 반복"하는 **탐욕(greedy) 접근**을 시도하고 있다. 이 접근은 다음 이유로 본 문제에 적합하지 않다.

1. 가장 큰 단일 거래의 매수일과 매도일을 그대로 잠그면, "그 단일 거래의 구간을 두 개의 더 좁은 거래로 쪼개는" 해가 최적인 경우를 표현할 수 없다. 반례: $[1, 5, 3, 6]$, $k=2$의 DP 최적해는 $(5-1)+(6-3)=7$이지만, 탐욕은 단일 최대 $6-1=5$를 먼저 잡고 day 0과 day 3을 제거하므로 남은 $[5, 3]$에서 $0$만 더해 총합 $5$가 된다. 7을 얻으려면 day 0과 day 3을 서로 다른 거래의 양 끝점으로 분배해야 한다.
2. "배열에서 매수일/매도일을 제거"하는 조작은 시간순 제약 $i_1 \leq j_1 < i_2$를 보존하지 않는다. 거래는 시간순으로 분리되어야 하며, 단순한 인덱스 삭제로는 표현되지 않는다.

따라서 알고리즘적 방향 자체를 DP로 교체해야 한다.

실패한 케이스:

|------|------|------|

| `k=2, [3,2,6,5,0,3]` | 7 | 8 |
| `k=100, [1,2,3,4,5]` | 4 | 400 |
| `k=100, [0,10000]` | 10000 | 1000000 |

원인은 세 가지 결함이 결합된 결과이다.

### 결함 A — `Array.prototype.splice`의 반환값 오해

```ts
prices = prices.splice(minPIdx, 1);
prices = prices.splice(maxProfitQIdx, 1);
```

`splice(start, deleteCount)`는 **원본 배열을 제자리에서 변형**하면서 **제거된 원소들의 배열**을 반환한다. 따라서 `prices = prices.splice(...)`는 "제거된 원소 1개로 이루어진 배열"을 `prices`에 대입한다. 첫 외부 반복 이후 `prices`는 길이 1 이하가 되며, 이후 내부 루프(`i < prices.length`)는 사실상 실행되지 않는다.

### 결함 B — 상태 변수가 외부 루프에서 재초기화되지 않음

```ts
let minPrice = prices[0]!;
let maxProfit = 0;
let minPIdx = 0;
let maxProfitQIdx = 0;

for (let j = 0; j < k; j++) { ... }
```

`minPrice`, `maxProfit`, `minPIdx`, `maxProfitQIdx`는 외부 루프 밖에서 선언되어 있어 다음 반복에서도 이전 값이 유지된다. 결함 A로 인해 내부 루프가 더 이상 갱신되지 않더라도, `maxProfit`이 직전 반복의 값을 유지하므로 `totalProfit += maxProfit`이 매 반복마다 같은 값을 더한다.

- 입력 `k=100, [0, 10000]`: 첫 반복에서 `maxProfit=10000`, 이후 99회 동안 같은 값이 더해져 총 $10000 \times 100 = 1{,}000{,}000$.
- 입력 `k=100, [1,2,3,4,5]`: 첫 반복 `maxProfit=4`, 이후 99회 누적 → $400$.

### 결함 C — 탐욕 전략 자체의 부정확성

결함 A, B를 수정해도 일반적인 입력에서 정답을 보장하지 못한다. 반례 $[1, 5, 3, 6]$, $k=2$에서 탐욕은 단일 최대 $6-1=5$를 먼저 확정하고 day 0, day 3을 제거하므로 남은 $[5, 3]$에서 추가 이익을 얻지 못해 총합 $5$를 반환한다. DP 최적해는 $(5-1)+(6-3)=7$이며, 이를 위해서는 "단일 최대 거래의 구간을 두 거래로 쪼개는" 탐색이 필요한데 탐욕은 이를 표현하지 못한다.

`[3,2,6,5,0,3]`처럼 단일 최대 거래의 매수일/매도일과 두 번째 거래의 양 끝점이 서로 겹치지 않는 입력에서는 탐욕과 DP의 답이 우연히 일치한다. 그러나 위 반례처럼 쪼개기가 이득인 입력에서는 탐욕이 실패한다.

`else if` 분기 구조도 부정확성에 기여한다. 새 최저가가 갱신된 직후 같은 인덱스에서 이익 계산을 건너뛴 채로 다음 단계로 넘어가도 1회 거래 문제에서는 문제없지만(매수일 = 매도일이면 이익 0), 본 코드는 `minPIdx`만 갱신하고 `maxProfit`은 직전 값을 유지하므로 시간순 정합성이 깨진다.

## 3. 최적화 분석

### 3.1. 올바른 알고리즘 채택 — DP로의 교체

방향성이 어긋나 있으므로, 최적화 이전에 알고리즘 자체를 교체해야 한다. 가이드의 DP를 채택하면 다음과 같다.

```ts
export function bestTimeToBuyAndSellStockK(
  k: number,
  prices: number[],
): number {
  const N = prices.length;
  if (k === 0 || N < 2) return 0;

  // k가 N/2 이상이면 사실상 무제한 거래 → 모든 상승분 합
  if (k >= Math.floor(N / 2)) {
    let total = 0;
    for (let i = 1; i < N; i++) {
      if (prices[i]! > prices[i - 1]!) {
        total += prices[i]! - prices[i - 1]!;
      }
    }
    return total;
  }

  const buy = new Array<number>(k + 1).fill(-Infinity);
  const sell = new Array<number>(k + 1).fill(0);

  for (let i = 0; i < N; i++) {
    const price = prices[i]!;
    for (let t = 1; t <= k; t++) {
      buy[t] = Math.max(buy[t]!, sell[t - 1]! - price);
      sell[t] = Math.max(sell[t]!, buy[t]! + price);
    }
  }

  return sell[k]!;
}
```

- 시간복잡도 $O(Nk)$, 공간복잡도 $O(k)$. $N \cdot k \leq 1{,}000 \times 100 = 10^5$ 이므로 성능 테스트(100ms)도 여유롭게 통과한다.
- `k >= N/2` 분기는 매일 매수·매도가 허용된 무제한 거래와 동치이므로 $O(N)$의 상승분 합산으로 즉시 답을 얻는다.

### 3.2. 핵심 아이디어 — 알고리즘 원형으로부터의 유도

#### 원형: 1회 거래의 좌측 최솟값 추적

1회 거래 문제(`bestTimeToBuyAndSellStock.ts`)는 다음을 계산한다.

$$
\max_{0 \leq i \leq j < N}\bigl(prices[j] - prices[i]\bigr)
$$

상태 변수 $minP$가 $\min_{0 \leq i \leq j} prices[i]$를 추적하고, 매 $j$에서 $prices[j] - minP$의 최대를 갱신한다. 이는 "현재 보유 중일 때의 최대 이익"과 "오늘 매도했을 때의 최대 이익"을 한 변수씩으로 줄인 형태이다.

#### 1단계: 상태를 2-튜플로 명시

1회 거래의 두 변수 $minP,\ maxPrice$를 다음과 같이 재명명한다.

- $-minP$ = "한 번 매수만 마치고 아직 매도하지 않은 상태의 최대 누적 이익"
- $maxPrice$ = "한 번 매도까지 마친 상태의 최대 이익"

매수 직후의 상태에서는 $prices[i]$만큼 비용이 발생하므로 이익은 $-prices[i]$이다. 매도 직후의 상태에서는 $-prices[i^*] + prices[j]$로 이익이 누적된다. 두 상태를 각각

$$
buy^{(1)},\ sell^{(1)}
$$

로 부르면 점화식은:

$$
buy^{(1)} \leftarrow \max(buy^{(1)},\ -prices[i])
$$
$$
sell^{(1)} \leftarrow \max(sell^{(1)},\ buy^{(1)} + prices[i])
$$

(초기값 $buy^{(1)} = -\infty,\ sell^{(1)} = 0$.)

이 점화식은 1회 거래의 좌측 최솟값 추적과 수학적으로 동치이다. $buy^{(1)} = -minP$이고 $sell^{(1)} = maxPrice$이다.

#### 2단계: 거래 횟수 차원 도입

거래를 두 번 이상 허용하려면 "지금까지 몇 번째 거래의 어느 단계에 있는가"를 상태에 추가해야 한다. $t \in \{1, 2, \ldots, k\}$를 거래 인덱스로 두고,

- $buy[t]$ = $t$번째 매수까지 마친 상태의 최대 누적 이익
- $sell[t]$ = $t$번째 매도까지 마친 상태의 최대 누적 이익

으로 확장한다. $t$번째 매수를 하려면 직전에 $(t-1)$번째 매도까지 끝나 있어야 하므로:

$$
buy[t] \leftarrow \max(buy[t],\ sell[t-1] - prices[i])
$$

$t$번째 매도는 $t$번째 매수 이후에 일어나므로:

$$
sell[t] \leftarrow \max(sell[t],\ buy[t] + prices[i])
$$

초기값은 $buy[t] = -\infty$ (아직 매수가 일어나지 않음), $sell[0] = 0$ (0회 거래의 이익은 0).

#### 3단계: 시간순 보존을 통한 정합성

내부 루프에서 $buy[t]$를 먼저 갱신한 다음 즉시 $sell[t]$를 갱신한다. 이는 "같은 날 $i$에 $t$번째 매수와 매도가 동시에 일어나면 이익은 $0$"이라는 자명한 경우만 추가될 뿐이므로 정답을 망치지 않는다. 한편 $sell[t-1]$이 $buy[t]$ 갱신 이전에 참조되도록 보장하려면 $t$를 $1$부터 $k$까지 오름차순으로 순회한다. 이 순서로 인해 각 $t$의 $sell[t-1]$은 같은 $i$ 내에서 아직 갱신되지 않은 이전 상태를 유지하며, 시간순 제약 $i_1 \leq j_1 < i_2 \leq j_2$가 보존된다.

#### 4단계: 차분 부분합 관점에서의 재해석

이익을 차분의 망원합으로 풀면:

$$
prices[j] - prices[i] = \sum_{m=i+1}^{j} (prices[m] - prices[m-1])
$$

$k$개의 비중첩 구간 $[i_m+1, j_m]$의 합을 최대화하는 문제는 "차분 배열 $D[m] = prices[m] - prices[m-1]$에서 $k$개의 비중첩 부분배열 합의 최대"이다. 이는 Kadane의 다중 부분합 일반화에 해당하고, 본 DP는 그 일반화를 "보유 여부"라는 이진 차원으로 인코딩한 형태이다.

특수 케이스 $k \geq \lfloor N/2 \rfloor$에서는 $D[m] > 0$인 모든 항목을 따로 골라 합산하는 것이 최적이다. $D$의 음수 항목은 비중첩 조건을 추가 비용 없이 회피 가능하므로, 양수 항목의 합이 곧 답이다.

### 3.3. 남은 버그(미수정)

본 보고서 작성 시점의 `bestTimeToBuyAndSellStockK.ts` 코드는 결함 A·B·C가 모두 남아 있다. 위 권장 코드를 적용하면 11개 테스트 모두 통과할 것으로 예상되며, 성능 테스트(N=1000, k=100, 100ms)도 $O(Nk) = 10^5$ 연산이므로 여유롭게 통과한다.

## 4. 결론

- **방향성**: 현재 솔루션은 가이드의 DP 접근을 따르지 않고, 단일 최대 거래를 $k$번 반복하는 탐욕 변형을 시도한다. 이 방향은 본 문제에 부적합하다.
- **정확성**: 11개 테스트 중 4개 실패. 결함 A(`splice` 반환값 오용), 결함 B(외부 루프 외부의 상태 비초기화), 결함 C(탐욕 전략 자체의 부정확성)가 결합되어 NaN/과대 추정 결과를 만든다.
- **최적화**: 알고리즘 교체가 선행되어야 한다. 가이드의 $O(Nk)$ DP와 $k \geq \lfloor N/2 \rfloor$ 특수 케이스 분기를 적용하면 모든 제약을 만족한다.
