# guide-for-problem 스킬 (가이드 집필 워크플로우)

`src/$ARGUMENTS.ts`의 주석(문제)·함수·테스트를 입력으로, 문제를 푸는 **깊은 해설서**
`src/$ARGUMENTS-guide.mdx`를 작성한다.

집필은 `~/centurio1987.github.io`의 기술 글 워크플로우(**리서치 → 집필 → 품질 게이트웨이 → 발행**)를
알고리즘 가이드에 맞게 옮겨 온 **4단계 파이프라인**으로 진행한다. 각 단계의 사양은 아래에 그대로 반영했다.
단, **가이드의 구성 항목**(성능 목표 예측 / 목표 함수 / 핵심 아이디어 / 시뮬레이션 / 수도 코드와 Activity
Diagram)은 기존 그대로 준수한다 — 워크플로우만 재정의하고 산출물의 골격은 바꾸지 않는다.

```
[1. 리서치]  src/<name>.ts 주석·함수·테스트 + 알고리즘 원형/복잡도 사실 수집 → 가이드 앵글(핵심 아이디어 유도의 출발점)
  → [2. 집필]  src/<name>-guide.mdx (canvas 구조로 집필 + 시뮬 위탁 + 외부 검토: 누락·모순·사실성)
  → [3. 품질 게이트]  QUALITY_CHECKLIST.md로 채점 → 보완 → 재채점 (≤3회, PASS까지)
  → [4. 발행]  frontmatter·빌드·시뮬 확정 → 가이드 인덱스 등록(generate-guides-index)
```

## 이 가이드가 반드시 충족해야 하는 것 (ORDER 원칙)

집필 내내 아래를 기준으로 삼는다. 어기면 가이드를 고친다. (블로그 워크플로우의 집필 원칙을 알고리즘
가이드에 맞춰 옮긴 것이다.)

- **독자 = 고객.** 독자가 끝까지 이해하지 못하면 실패다. "어떻게든 이해시킨다"는 태도로 쓴다.
- **점진적 공개.** 쉬운 것 → 어려운 것 순서로 쌓는다. 뒤 내용이 앞에서 설명하지 않은 전제에 기대지 않는다.
- **해부학적 깊이.** 표면 설명에서 멈추지 않는다. "왜 그렇게 동작하는가"까지 파고든다. 무언가를 주장할
  때는 근거(증명·복잡도 유도·반례)를 가급적 동반한다.
- **유도 생략 없음.** 원천 아이디어·알고리즘을 먼저 설명하고, **거기서 솔루션으로 유도하는 과정을 생략 없이**
  제공한다. 원형 알고리즘이 있으면 원형 → 관찰 → 개선의 전개를 단계별로 보인다.
- **정식 표기.** 핵심 아이디어는 초등학생도 따라올 만큼 꼼꼼하게, 단 **비유나 묘사 없이 정식 표기법과 용어만**
  사용한다(수식 $…$ 포함). 이는 `analyze-solution`의 서술 원칙과 일치한다.
- **다관점.** 한 개념을 정의·수식·트레이스·다이어그램·시뮬레이션 등 여러 각도로 비춘다.
- **한국어 본문 + 기술 용어 원어** 유지(예: pivot, partition, $O(N \log N)$).

## 1단계 — 리서치 (research)

> 블로그 `research` 스킬의 대응 단계. 집필 에이전트가 다시 찾지 않아도 될 만큼 **사실을 먼저 굳힌다.**
> 알고리즘 가이드의 "사실"은 문제 정의·실제 구현 동작·표준 알고리즘 지식(원형·복잡도·정확성)이다.

1. **입력 정독**: `Read`로 `src/$ARGUMENTS.ts` 전체(특히 문제 주석)와 함수 구현을 읽는다. `-problem.md`가
   있으면 함께 읽어 제약·예시를 파악한다.
2. **실제 동작 확인**: `bun test src/$ARGUMENTS.test.ts 2>&1; true`로 현재 구현의 통과/실패와 동작을
   확인한다. 가이드의 트레이스·시뮬레이션·복잡도 서술은 **머릿속 추정이 아니라 이 실제 동작**에 근거한다.
3. **알고리즘 사실 수집**: 이 문제가 기대하는 목표 복잡도, 원형 알고리즘, 정확성 근거(불변식·증명 골자)를
   정리한다. 표준 사실(예: 비교 정렬 하한 $O(N \log N)$)은 단정하지 말고 근거를 확인한다 — 불확실하면
   `WebSearch`/`WebFetch`로 1차 자료(교과서·표준·공식 문서)를 확인하고 **검증한 사실만** 쓴다.
   - **skip 조건**: 원형·복잡도가 자명한 표준 알고리즘이면 외부 조사는 건너뛸 수 있다(블로그의 "외부 자료
     불필요 시 skip"과 동일). 단 건너뛴 사실도 본문에서 근거 없이 단정하지는 않는다.
4. **앵글 확정**: 이 가이드의 핵심 주장 = "**원천 아이디어 → 솔루션 유도의 경로**"를 한 문단으로 정리한다.
   이것이 2단계 집필의 입력이며 `## 핵심 아이디어` 섹션의 뼈대가 된다. 부족한 자료는 open question으로
   남겨 집필 시 주의한다.

## 2단계 — 집필 (writing)

> 블로그 `tech-deepdive` 대응 단계. 산출물은 `src/$ARGUMENTS-guide.mdx` — `src/$ARGUMENTS.ts`와 나란히 둔다.

### 2.1 구성 항목 (canvas 구조 — 변경 금지)

`solving-problem-canvas.md`의 골격을 그대로 따른다. 산문 4개 섹션 + 시뮬레이션:

1. **성능 목표 예측** — 입력 규모·제약에서 요구되는 목표 복잡도를 예측한다.
2. **목표 함수** — 풀어야 하는 함수의 시그니처·입출력 계약을 명시한다.
3. **핵심 아이디어** — 원천 아이디어/알고리즘을 먼저 설명하고, 솔루션으로 **유도하는 과정을 생략 없이**
   전개한다(원형 알고리즘이 있으면 원형에서 유도). 정식 표기·용어만 사용, 비유 금지.
4. **시뮬레이션** — 2.2 참조.
5. **수도 코드와 Activity Diagram** — 최종 알고리즘을 의사코드와 흐름으로 정리한다.

MDX에서 markdown·수식(`$…$`)·코드블록은 그대로 유효하다. 단 `{`, `<` 등 JSX와 충돌하는 원문은 코드블록
안에 두거나 이스케이프한다.

### 2.2 시뮬레이션 위탁 (`#guide-sim`)

알고리즘 전개를 단계별로 조작·재생하는 React 시뮬레이션을 `## 핵심 아이디어` 뒤 `## 시뮬레이션` 섹션에 넣되,
**문제별 JSX 컴포넌트/Hook을 직접 작성하지 않고** 공용 모듈에 위탁한다(블로그가 시뮬을 `react-sim`에
위탁하는 것과 같은 경계).

- MDX 상단에 `import { AlgorithmSimulation } from "#guide-sim";` 한 줄만 추가한다. `#guide-sim` 외 import 금지.
- 적합한 프리셋 `view`를 고르고, 작은 고정 입력의 **실제 실행 단계**를 선언형 `steps` 데이터로 나열한다.
- 프레임 스키마·프리셋 목록·예시는 `simulation-scaffold.md`를 따른다.
- **정확성**: `steps`는 1단계에서 확인한 원본 함수의 **실제 실행**을 따른다. 사용한 고정 입력과 그 실제
  반환값을 본문에 명시하고, 마지막 프레임의 결과 상태가 그 값과 일치하게 한다.
- 시뮬레이션 호출부 바로 위에 정적 뷰어용 안내 한 줄("대화형 시뮬레이션은 MDX 런타임에서 표시됩니다.")을 둔다.

### 2.3 외부 검토 (누락·모순·사실성)

집필을 마치면 외부 모델(codex/agy)에 검토를 받아 보강한다. 블로그 `tech-deepdive`의 외부 검토 단계를
알고리즘 가이드에 맞춘 것이다(축: 누락/생략, 모순/충돌, **사실 정확성 — 복잡도·정확성 증명·표준 알고리즘**).

```bash
bash .claude/skills/guide-for-problem/scripts/review-guide.sh "src/$ARGUMENTS-guide.mdx"
```

- 스크립트 stdout을 **요약하지 말고 그대로** 사용자에게 보여준다(모델이 짚은 표현이 핵심).
- 종료코드 2(둘 다 `[skip]`/실패)면 외부 검토 미통과다 — 가이드를 "완성"으로 보고하지 말고 원인(미설치/미인증:
  `codex login`, 터미널에서 `agy` 1회 실행)을 안내하고 재시도 여부를 묻는다. 한쪽만 성공하면 진행하되 알린다.
- 지적 중 **실제로 유효한 것**을 사용자와 함께 골라(외부 모델은 이 가이드의 의도/맥락을 모른다) 본문을 보강한다.

## 3단계 — 품질 게이트웨이 (quality gate)

> 블로그 `quality-gate` 대응 단계. `assets/QUALITY_CHECKLIST.md`로 **PASS/FAIL을 판정**하고, 미통과면
> **보완 → 재채점**을 PASS까지(최대 3라운드) 반복한다.

1. `assets/QUALITY_CHECKLIST.md`의 모든 항목을 순서대로 **충족 / 미흡 / 해당없음** + **본문 인용 근거**로
   평가한다(인용 못 하면 미흡). 해당없음은 "왜 뺐는지 메모가 있는가"로 검증한다.
2. 판정 규칙: **PASS = MUST 전부 충족 · SHOULD 미흡 ≤ 2 · IF-APPLICABLE은 충족 또는 정당한 생략 메모.**
   그 외는 FAIL. 빌드 항목(G1)은 `bun run build`로 실제 확인한다. 사실성(E2/E3)은 필요 시 `WebSearch`로 확인.
3. **FAIL이면 보완 루프**: 미흡 항목마다 "어디·무엇·어떻게"의 보완 지시를 내고, 단순 문장 교정은 직접 `Edit`,
   **누락 섹션·유도 부족·시나리오 보강처럼 새 집필이 필요한 보완은 2단계로 되돌아가** 해당 섹션만 다시 쓴다.
   재채점하고 라운드를 올린다. **PASS면 종료**. 3라운드를 채워도 FAIL이면 **멈추고** 남은 미흡과 자동 보완이
   어려운 이유를 보고한다 — 기준을 낮춰 임의 통과시키지 않는다.
4. 호출 맥락:
   - **사용자 직접 호출**: 라운드 1 결과를 보여준 뒤 "(a) PASS까지 자동 보완 / (b) 항목 골라 보완 / (c) 여기까지"를 묻는다.
   - **파이프라인 안에서**: 루프를 자동으로 돌리되 라운드마다 무엇을 고쳤는지 남기고, PASS 전에는 발행으로 넘어가지 않는다.

## 4단계 — 발행 (publishing)

> 블로그 `publish-post` 대응 단계. 알고리즘 가이드는 별도 콘텐츠 컬렉션 발행 대신 **프로젝트 인덱스 등록**으로
> 발행을 마친다(사이드바 수동 등록 없음 — 블로그가 콘텐츠 컬렉션 자동 탐색인 것과 동일한 정신).

PASS한 뒤에만 진행한다.

1. **최종 점검**: 가이드가 `src/$ARGUMENTS-guide.mdx`에 있고, frontmatter(있다면)·import 경로·시뮬 컴포넌트가
   올바른지 확인한다. `bun run build`(또는 해당 빌드/타입 체크)가 통과하는지 마지막으로 확인한다.
2. **인덱스 등록**: `generate-guides-index` 스킬(또는 `./tools/generate-guides-index.sh`)을 실행해
   `문제_가이드_목록.md`에 새 가이드가 등록되게 한다.
3. **완료 보고**: 생성된 가이드 경로, 인덱스 등록 결과, 품질 게이트 최종 판정(PASS/라운드 수)을 한 번에 보고한다.

## 참조 파일

- `solving-problem-canvas.md` — 가이드 구성 항목(골격, 변경 금지)
- `simulation-scaffold.md` — 시뮬레이션 프레임 스키마·프리셋·작성 규칙(`#guide-sim` 위탁)
- `assets/QUALITY_CHECKLIST.md` — 3단계 품질 게이트 루브릭(MUST/SHOULD/IF-APPLICABLE)
- `scripts/review-guide.sh` — 2.3 외부 검토(codex/agy: 누락·모순·사실성)

## 주의

- **구성 항목은 바꾸지 않는다.** 워크플로우(4단계)만 재정의했고, 가이드 골격(성능/목표 함수/핵심 아이디어/
  시뮬레이션/수도코드+Activity Diagram)은 기존 그대로다.
- **자동 통과 금지**: 품질 게이트 3라운드를 다 써도 미통과면 그대로 보고한다.
- **외부 검토 미통과(둘 다 실패)** 상태를 "완성"으로 보고하지 않는다.
- 시뮬레이션은 `#guide-sim`에만 위탁한다 — 문제별 JSX/Hook 직접 작성·외부 라이브러리 import 금지.
- MDX 작성 후 빌드 통과를 확인한다(JSX 문법 오류 1건이 빌드 전체를 깰 수 있다).
