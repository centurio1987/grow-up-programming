# 가이드 품질 체크리스트 (Quality Rubric)

`guide-for-problem` 3단계(품질 게이트웨이)가 쓰는 루브릭이다. `~/centurio1987.github.io`의 기술 글
품질 체크리스트를 **알고리즘 가이드**에 맞춰 옮겨 왔다 — 항목은 가이드의 구성(성능 목표 예측 / 목표 함수 /
핵심 아이디어 / 시뮬레이션 / 수도 코드와 Activity Diagram)과 ORDER 원칙(독자=고객, 점진적 공개,
유도 생략 없음, 정확성)을 채점한다.

각 항목은 **충족 / 미흡 / 해당없음**으로 평가하고, 미흡이면 **어디서·왜·어떻게 고칠지** 한 줄 보완 지시를
남긴다. 평가는 반드시 **본문에서 근거를 인용**해서 한다(추측 금지). 인용이 안 나오면 그 항목은 미흡으로 본다.

---

## 분류와 가중치

- **필수(MUST)** — 하나라도 미흡이면 게이트 **미통과**. ORDER가 명시적으로 요구한 핵심.
- **권장(SHOULD)** — 미흡이 **2개 이하**면 통과 가능(보고에 남김). 3개 이상이면 미통과.
- **해당시(IF-APPLICABLE)** — 주제에 해당할 때만 평가. 뺐다면 "왜 뺐는지" 메모가 있어야 충족.

게이트 통과 조건: **MUST 전부 충족 + SHOULD 미흡 ≤ 2 + 해당시 항목은 충족 또는 정당한 생략 메모**.

---

## A. 독자 이해 (reader = customer)

- **[MUST] A1 점진적 공개** — 쉬운 개념 → 어려운 개념 순서로 쌓이는가. 뒤 내용이 앞에서 설명 안 한 전제에 기대지 않는가.
- **[MUST] A2 용어·표기 풀어쓰기** — 핵심 용어·기호가 첫 등장 시 정의되는가(비전문가가 막힐 미정의 용어가 없는가).
- **[SHOULD] A3 정식 표기 준수** — 핵심 아이디어 설명이 비유·묘사 없이 정식 표기법과 용어만 쓰는가(꼼꼼하되 느슨한 통념에 기대지 않는가).

## B. 핵심 아이디어 — 유도의 깊이 (anatomical depth)

- **[MUST] B1 원천 아이디어 선행** — 풀이를 던지기 전에 원천이 되는 아이디어/알고리즘을 먼저 설명하는가.
- **[MUST] B2 유도 생략 없음** — 원천에서 솔루션으로 가는 과정을 **단계별로 생략 없이** 전개하는가(원형이 있으면 원형 → 관찰 → 개선).
- **[MUST] B3 "왜"의 존재** — 핵심 주장마다 근거(불변식·증명 골자·복잡도 유도·반례)가 붙는가. 표면 설명에서 멈추지 않는가.
- **[SHOULD] B4 트레이드오프** — 대안·한계·비용을 짚는가(가능한 다른 접근 대비 이 풀이의 위치).

## C. 완전성 — 구성 항목 (completeness, canvas 골격)

- **[MUST] C1 성능 목표 예측** — 입력 규모·제약에서 요구되는 목표 복잡도를 예측하는 절이 있는가.
- **[MUST] C2 목표 함수** — 풀어야 하는 함수의 시그니처·입출력 계약이 명시돼 있는가.
- **[MUST] C3 핵심 아이디어** — 위 B축을 만족하는 핵심 아이디어 섹션이 있는가.
- **[MUST] C4 수도 코드와 Activity Diagram** — 최종 알고리즘이 의사코드와 흐름(Activity Diagram)으로 정리돼 있는가.
- **[SHOULD] C5 마무리/요약** — 핵심 재진술 + 복잡도 결론(목표 달성 여부)이 있는가.

## D. 다관점 (multi-perspective)

- **[MUST] D1 다각 조명** — 최소 한 개념을 **2개 이상 관점**(정의/수식 + 트레이스, 원리 + 시뮬 등)으로 비추는가.
- **[IF-APPLICABLE] D2 트레이스 예시** — 작은 고정 입력에 대한 단계별 손 추적(trace)이 있는가(없으면 생략 메모).
- **[IF-APPLICABLE] D3 다이어그램** — 구조/흐름이 복잡하면 그림 또는 Activity Diagram이 있는가.
- **[IF-APPLICABLE] D4 시뮬레이션** — "조작하며 관찰"해야 이해되는 동적 전개가 있으면 시뮬이 있는가(`#guide-sim` import + 선언형 `steps`, 본문에 조작/관찰 안내).
- **[IF-APPLICABLE] D5 구현/코드** — 코드 인용이 맥락 없이 먼저 나오지 않고, 줄/블록 의미가 설명되는가.

## E. 정확성·근거 (correctness)

- **[MUST] E1 모순 없음** — 앞뒤가 충돌하지 않는가(복잡도·정의·결론 일관).
- **[MUST] E2 사실 정확성** — 복잡도·정확성 증명·표준 알고리즘 설명이 정확한가. 근거가 있거나 검증 가능한가.
- **[MUST] E3 실제 동작 일치** — 본문의 트레이스·복잡도·시뮬 결과가 `src/<name>.ts`의 **실제 실행**(테스트 결과 포함)과 일치하는가(추정이 아닌가).
- **[SHOULD] E4 시대성** — 폐기됐거나 낡은 정보가 현재처럼 서술되지 않았는가.

## F. 문체·몰입 (게이트 수준)

- **[MUST] F1 비-덤프 문체** — 스펙·단계를 건조하게 나열하지 않고 "왜 이렇게 푸는지/무슨 문제를 푸는지"를 먼저 거는가.
- **[SHOULD] F2 도입 훅** — 첫 1~2문단(또는 성능 목표 예측)이 풀이의 동기를 끌어당기는가.

## G. 기술 건전성 (MDX/빌드)

- **[MUST] G1 빌드 통과** — `bun run build`가 통과하는가(JSX 문법·import 경로·frontmatter).
- **[IF-APPLICABLE] G2 시뮬 위탁 규약** — 시뮬이 있으면 `#guide-sim` import + 선언형 `steps`만 쓰고, 문제별 JSX/Hook·외부 라이브러리 import가 없는가.
- **[IF-APPLICABLE] G3 시뮬 정확성** — 시뮬의 고정 입력·마지막 프레임 결과가 본문에 명시된 실제 반환값과 일치하는가.

---

## 점수 산정 (보고용)

- MUST: `통과 X / 전체 Y` (하나라도 미흡 → 게이트 미통과)
- SHOULD: 미흡 개수 (≤2 통과)
- IF-APPLICABLE: 충족 / 정당한 생략 / 미흡

최종: **PASS** 또는 **FAIL — 보완 필요**. FAIL이면 "[항목코드] 인용 → 문제 → 보완" 형식의 보완 지시를 함께 낸다.
