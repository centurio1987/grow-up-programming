# 문제 품질 체크리스트 (Quality Rubric)

`gen-problem` 3단계(품질 게이트웨이)가 쓰는 루브릭이다. `~/centurio1987.github.io`의 기술 글 품질
체크리스트를 **코드 테스트 문제 집필**에 맞춰 옮겨 왔다 — 항목은 문제의 구성(함수 인터페이스 / 제약 조건 /
문제 상세 / 예시)과 ORDER 원칙(독자=고객, 빠른 이해, 스토리, 모호성 제거, **풀이 힌트 금지**, 검증 가능성)을
채점한다.

각 항목은 **충족 / 미흡 / 해당없음**으로 평가하고, 미흡이면 **어디서·왜·어떻게 고칠지** 한 줄 보완 지시를
남긴다. 평가는 반드시 **본문에서 근거를 인용**해서 한다(추측 금지). 인용이 안 나오면 그 항목은 미흡으로 본다.

---

## 분류와 가중치

- **필수(MUST)** — 하나라도 미흡이면 게이트 **미통과**. ORDER가 명시적으로 요구한 핵심.
- **권장(SHOULD)** — 미흡이 **2개 이하**면 통과 가능(보고에 남김). 3개 이상이면 미통과.
- **해당시(IF-APPLICABLE)** — 주제에 해당할 때만 평가. 뺐다면 "왜 뺐는지" 메모가 있어야 충족.

게이트 통과 조건: **MUST 전부 충족 + SHOULD 미흡 ≤ 2 + 해당시 항목은 충족 또는 정당한 생략 메모**.

---

## A. 독자 이해 (reader = customer)

- **[MUST] A1 빠른 정체 파악** — 첫 1~2문단(스토리·한 줄 요약)에서 "이 함수가 무엇을 입력받아 무엇을 반환하는지"가 잡히는가.
- **[MUST] A2 모호성 제거** — 모호어("적당히", "큰 수", "빠르게")가 없고, 한 가지 해석만 남도록 규칙이 못 박혀 있는가.
- **[MUST] A3 용어·표기 풀어쓰기** — 핵심 용어·기호·단위가 첫 등장 시 정의되는가(인덱스 0/1 기준, 정렬 안정성 등 포함).
- **[SHOULD] A4 점진적 공개** — 쉬운 요약 → 정식 계약 → 세부 제약 순으로 쌓이는가(앞에서 안 깐 전제에 기대지 않는가).

## B. 스토리·동기 (story, ORDER 명시 요구)

- **[MUST] B1 스토리 존재** — 간단한 상황/맥락으로 문제에 동기를 부여하는 스토리가 있는가.
- **[MUST] B2 스토리 ↔ 계약 정합** — 스토리가 말하는 목표와 함수 시그니처/반환이 일치하는가(스토리가 본문과 따로 놀지 않는가).
- **[SHOULD] B3 군더더기 없음** — 스토리가 이해를 돕는 선에서 간결한가(필요 이상으로 길거나 산만하지 않은가).

## C. 완전성 — 구성 항목 (completeness, 문제 골격)

- **[MUST] C1 함수 인터페이스** — 시그니처(이름·파라미터·반환 타입)가 코드블록으로 명시돼 있고, 파라미터명에 언더바 prefix가 없는가.
- **[MUST] C2 제약 조건** — 입력 크기·값 범위·시간/공간 상한이 **정량적으로** 명시돼 있는가.
- **[MUST] C3 문제 상세** — 계산/변환 규칙과 **경계·예외 동작**(빈 입력·동점·음수·오버플로 경계 등)이 빠짐없이 규정돼 있는가.
- **[MUST] C4 예시** — (입력→출력) 예시가 2개 이상 있고, **경계 케이스 예시**가 최소 1개 포함되는가.
- **[SHOULD] C5 예시 설명** — 각 예시에 "왜 그 출력인지"가 한 줄로 붙어 있는가(규칙 적용만, 풀이 절차 아님).

## D. 풀이 힌트 금지 (no-solution-leak, ORDER 명시 금기)

- **[MUST] D1 접근법 미암시** — 본문(스토리 포함)이 알고리즘·자료구조·풀이 절차·복잡도 트릭을 암시하지 않는가.
- **[MUST] D2 예시 설명의 절제** — 예시 해설이 *규칙의 적용*만 보이고 *어떻게 푸는지*를 보이지 않는가.

## E. 정확성·검증 가능성 (correctness)

- **[MUST] E1 모순 없음** — 제약·규칙·예시·시그니처가 서로 충돌하지 않는가.
- **[MUST] E2 예시 정확성** — 모든 예시의 출력이 본문 규칙(과 실제 정답 동작)과 일치하는가. 필요 시 함수를 실제로 돌려 확인.
- **[MUST] E3 글 ↔ 테스트 정합** — 본문 예시의 기대값이 `src/<name>.test.ts`의 기대값과 일치하는가.
- **[SHOULD] E4 검증 가능성** — 본문이 요구하는 모든 동작이 테스트로 표현 가능한가(서술만 있고 검증 불가한 요구가 없는가).

## F. 테스트 커버리지 (test coverage, `checklist.md` 연동)

- **[MUST] F1 바운더리 테스트** — 입력 경계(빈/단일/최소/최대)에 대한 테스트가 있는가.
- **[MUST] F2 엣지 케이스** — 동점·음수·중복·특수값 등 문제 상세가 규정한 예외가 테스트로 덮이는가.
- **[SHOULD] F3 성능 테스트** — 제약 상한 입력에 대한 성능 테스트가 있는가(CPU 1초 ≈ $10^8$ 연산 가정).
- **[SHOULD] F4 구현 비의존** — 테스트가 풀이 구현 세부가 아니라 함수의 외부 계약만 검증하는가.

## G. 기술 건전성 (build/test)

- **[MUST] G1 테스트 통과** — 구현이 있을 때 `bun test src/<name>.test.ts`가 통과하는가.
- **[IF-APPLICABLE] G2 MDX 빌드** — `.mdx`로 작성했으면 `bun run build`(또는 타입 체크)가 통과하는가(JSX 문법·import 경로).
- **[IF-APPLICABLE] G3 JSX 절제** — `.mdx`를 썼다면 이해에 실제로 기여하는가(억지 컴포넌트가 아닌가). 아니면 `.md`로 충분하다는 메모가 있는가.

---

## 점수 산정 (보고용)

- MUST: `통과 X / 전체 Y` (하나라도 미흡 → 게이트 미통과)
- SHOULD: 미흡 개수 (≤2 통과)
- IF-APPLICABLE: 충족 / 정당한 생략 / 미흡

최종: **PASS** 또는 **FAIL — 보완 필요**. FAIL이면 "[항목코드] 인용 → 문제 → 보완" 형식의 보완 지시를 함께 낸다.
