# gen-problem 스킬 (문제 집필 워크플로우)

`src/$ARGUMENTS.ts`의 주석(미완성 문제 단서)·함수 시그니처를 입력으로, 독자가 **쉽고 빠르게 이해할 수 있는
풍성한 문제 문서** `src/$ARGUMENTS-problem.md`(스토리·예시·다이어그램이 필요해 JSX를 쓰면 `.mdx`)와,
그 문제를 **정확히 반영하고 충분히 커버하는 테스트 코드** `src/$ARGUMENTS.test.ts`를 작성한다.

집필은 `~/centurio1987.github.io`의 기술 글 워크플로우(**리서치 → 집필 → 품질 게이트웨이 → 발행**)를
**문제 집필**에 맞게 옮겨 온 **4단계 파이프라인**으로 진행한다. 각 단계의 사양은 아래에 그대로 반영했다.
단, **문제의 구성 항목**(함수 인터페이스 / 제약 조건 / 문제 상세 / 예시)과 **기존 조건**(파라미터명 언더바
prefix 금지, **풀이 힌트 미포함**)은 그대로 준수한다 — 워크플로우만 재정의하고 산출물의 골격·금기는 바꾸지 않는다.

```
[1. 리서치]  src/<name>.ts 주석·시그니처 + 문제 도메인/제약/정답 동작 사실 수집 → 문제 앵글(무엇을 묻는 문제인가)
  → [2. 집필]  src/<name>-problem.md(/.mdx) (스토리로 동기 부여 + 구성 항목 채움) + src/<name>.test.ts (외부 검토: 모호·모순·예시정확성·힌트누출)
  → [3. 품질 게이트]  PROBLEM_QUALITY_CHECKLIST.md로 채점 → 보완 → 재채점 (≤3회, PASS까지)
  → [4. 발행]  빌드·테스트 확인 → (해당 시) 인덱스/목록 등록
```

## 이 문제가 반드시 충족해야 하는 것 (ORDER 원칙)

집필 내내 아래를 기준으로 삼는다. 어기면 문제를 고친다. (블로그 워크플로우의 집필 원칙을 **문제 집필**에 맞춰
옮긴 것이다 — 핵심 변형: 우리는 *풀이*가 아니라 *문제 자체*를 이해시킨다.)

- **독자 = 고객.** 독자가 문제를 끝까지 이해하지 못하면 실패다. "독자가 이해해야 돈을 받는다"는 태도로,
  무엇을 입력받아 무엇을 돌려줘야 하는지 **오해의 여지 없이** 적극적으로 이해시킨다.
- **빠른 이해 우선.** 처음 1~2문단(스토리·한 줄 요약)에서 "이 문제가 무엇을 묻는지"가 바로 잡히게 한다.
  세부 제약은 그다음에 쌓는다(점진적 공개).
- **스토리로 동기 부여.** 문제에 **간단한 스토리**(현실적 상황·등장인물·목표)를 입혀 추상적 시그니처에
  맥락을 준다. 스토리는 *왜 이걸 계산하는지*를 알려주는 장치이지 **풀이 힌트가 아니다** — 알고리즘·자료구조·
  접근법을 암시하지 않는다.
- **풍성하되 군더더기 없이.** 제약·예시·경계 상황을 충분히 제공하되, 한 정보를 두 번 말하거나 풀이를
  흘리지 않는다.
- **정식 표기.** 입출력 계약·제약은 비유에 기대지 말고 **정식 표기법과 용어**(시그니처·타입·수식 $…$·복잡도
  상한)로 못 박는다. 모호어("적당히", "큰 수", "빠르게")는 정량 제약으로 바꾼다.
- **다관점.** 같은 규칙을 산문 정의 + 예시(입력→출력) + 경계/예외 사례로 여러 각도에서 비춰 한 해석만 남긴다.
- **검증 가능성.** 본문이 요구하는 모든 동작은 테스트로 표현 가능해야 한다. 글과 테스트가 같은 계약을 말한다.
- **한국어 본문 + 기술 용어 원어** 유지(예: array, in-place, $O(N \log N)$).

## 1단계 — 리서치 (research)

> 블로그 `research` 스킬의 대응 단계. 집필 에이전트가 다시 찾지 않아도 될 만큼 **사실을 먼저 굳힌다.**
> 문제 집필의 "사실"은 ① 주석에 적힌 의도, ② 함수 시그니처가 강제하는 입출력 타입, ③ 도메인의 표준 정의·
> 제약(자료형 한계·시간 복잡도 통념), ④ **정답 동작**(작은 입력에서 무엇이 올바른 출력인가)이다.

1. **입력 정독**: `Read`로 `src/$ARGUMENTS.ts` 전체(특히 문제 주석)와 함수 시그니처를 읽는다. 주석이 말하는
   *의도*와 시그니처가 강제하는 *타입·반환*을 정리한다. 같은 폴더의 기존 `-guide.mdx`/구현이 있으면 참고하되,
   **풀이는 문제 문서로 새지 않게** 분리해 둔다.
2. **정답 동작 확인**: 구현이 이미 있으면 `bun test src/$ARGUMENTS.test.ts 2>&1; true` 또는 작은 입력을
   직접 돌려 **올바른 출력**을 확인한다. 예시·경계 케이스의 기대값은 **머릿속 추정이 아니라 이 실제 동작**에
   근거한다. 구현이 없으면 손으로 작은 입력 2~3개의 정답을 계산해 적어 둔다.
3. **도메인 사실 수집**: 문제가 다루는 도메인의 표준 정의·단위·제약(예: 정수 범위, 인덱스 0/1 기준, 안정성)
   을 정리한다. 표준 사실이 불확실하면 `WebSearch`/`WebFetch`로 1차 자료를 확인하고 **검증한 사실만** 쓴다.
   - **skip 조건**: 도메인이 자명하면(기초 배열·정렬 등) 외부 조사는 건너뛸 수 있다. 단 건너뛴 사실도 근거
     없이 단정하지 않는다.
4. **앵글 확정**: 이 문제의 한 줄 정체 = "**누가, 무엇을 입력받아, 무엇을 반환해야 하는가**"와 그 스토리
   후크를 한 문단으로 정리한다. 이것이 2단계의 입력이며 스토리·문제 상세의 뼈대가 된다. **풀이 접근은 여기
   적지 않는다**(힌트 누출 방지). 모호한 지점은 open question으로 남겨 집필 시 정량 제약으로 못 박는다.

## 2단계 — 집필 (writing)

> 블로그 `tech-deepdive` 대응 단계. 산출물은 ① 문제 문서 `src/$ARGUMENTS-problem.md`(JSX 필요 시 `.mdx`),
> ② 테스트 코드 `src/$ARGUMENTS.test.ts` — 둘 다 `src/$ARGUMENTS.ts`와 나란히 둔다.

### 2.1 구성 항목 (문제 골격 — 변경 금지)

순서대로 채운다. **스토리 → 한 줄 요약**을 맨 앞에 두어 빠른 이해를 만든 뒤 정식 계약으로 못 박는다:

1. **스토리 & 한 줄 요약** — 간단한 상황/등장인물로 동기를 주고, 곧바로 "이 함수는 X를 받아 Y를 반환한다"는
   한 줄 요약으로 정체를 박는다. **풀이 암시 금지.**
2. **함수 인터페이스** — 풀어야 하는 함수의 시그니처(이름·파라미터·반환 타입)를 코드블록으로 명시한다.
   파라미터명은 **언더바 prefix를 쓰지 않는다.**
3. **제약 조건** — 입력 크기·값 범위·시간/공간 상한·특수 규칙을 **정량적으로** 나열한다(모호어 금지).
4. **문제 상세** — 무엇을 계산/변환해야 하는지 규칙을 산문으로 풀되, 경계·예외 동작(빈 입력, 동점, 음수,
   오버플로 경계 등)을 빠짐없이 규정한다.
5. **예시** — 최소 2~3개의 (입력 → 출력) 예시. 각 예시에 **왜 그 출력인지** 한 줄 설명을 붙이되 *규칙의
   적용*만 보이고 *풀이 절차*는 보이지 않는다. 경계 케이스 예시를 최소 1개 포함한다.

MDX에서 markdown·수식(`$…$`)·코드블록은 그대로 유효하다. 스토리 삽화나 작은 입출력 시각화가 이해를 크게
돕는 경우에 한해 `.mdx`로 저장하고 JSX를 쓸 수 있다(억지 컴포넌트 금지). `{`, `<` 등 JSX와 충돌하는 원문은
코드블록 안에 두거나 이스케이프한다.

### 2.2 테스트 코드 작성

문제 문서와 **같은 계약**을 말하는 테스트를 `src/$ARGUMENTS.test.ts`에 작성한다. 커버리지 기준은
`checklist.md`를 따른다.

- `import { test, expect } from "bun:test";` (Bun 테스트 API).
- **본문 예시 = 테스트 케이스**: 2.1의 모든 예시를 그대로 테스트로 옮겨, 글과 테스트의 기대값이 일치하게 한다.
- `checklist.md`의 세 축을 채운다: **바운더리 테스트** / **성능 테스트**(CPU 1초 ≈ $10^8$ 연산 가정으로
  제약 상한 입력) / **모든 엣지 케이스**. 각 기대값은 1단계에서 확인한 **정답 동작**에 근거한다(추정 금지).
- 테스트는 **풀이 구현에 의존하지 않고** 함수의 외부 계약만 검증한다.

### 2.3 외부 검토 (모호·모순·예시정확성·힌트누출)

집필을 마치면 외부 모델(codex/agy)에 검토를 받아 보강한다. 블로그 `tech-deepdive`의 외부 검토 단계를
**문제 집필**에 맞춘 것이다(축: 모호/미정의, 모순/충돌, **예시 정확성**, **풀이 힌트 누출**).

```bash
bash .claude/skills/gen-problem/scripts/review-problem.sh "src/$ARGUMENTS-problem.md"
```

- 스크립트 stdout을 **요약하지 말고 그대로** 사용자에게 보여준다(모델이 짚은 표현이 핵심).
- 종료코드 2(둘 다 `[skip]`/실패)면 외부 검토 미통과다 — 문제를 "완성"으로 보고하지 말고 원인(미설치/미인증:
  `codex login`, 터미널에서 `agy` 1회 실행)을 안내하고 재시도 여부를 묻는다. 한쪽만 성공하면 진행하되 알린다.
- 지적 중 **실제로 유효한 것**을 사용자와 함께 골라(외부 모델은 이 문제의 의도를 모른다) 본문·테스트를 보강한다.
  특히 "예시 출력이 규칙과 어긋난다", "이 서술이 풀이를 흘린다"는 지적은 최우선으로 처리한다.

## 3단계 — 품질 게이트웨이 (quality gate)

> 블로그 `quality-gate` 대응 단계. `assets/PROBLEM_QUALITY_CHECKLIST.md`로 **PASS/FAIL을 판정**하고,
> 미통과면 **보완 → 재채점**을 PASS까지(최대 3라운드) 반복한다.

1. `assets/PROBLEM_QUALITY_CHECKLIST.md`의 모든 항목을 순서대로 **충족 / 미흡 / 해당없음** + **본문 인용
   근거**로 평가한다(인용 못 하면 미흡). 해당없음은 "왜 뺐는지 메모가 있는가"로 검증한다.
2. 판정 규칙: **PASS = MUST 전부 충족 · SHOULD 미흡 ≤ 2 · IF-APPLICABLE은 충족 또는 정당한 생략 메모.**
   그 외는 FAIL. 빌드/테스트 항목은 `bun run build`·`bun test src/$ARGUMENTS.test.ts`로 실제 확인한다.
   예시 정확성은 필요 시 함수를 실제로 돌려 확인한다.
3. **FAIL이면 보완 루프**: 미흡 항목마다 "어디·무엇·어떻게"의 보완 지시를 내고, 단순 문장 교정은 직접 `Edit`,
   **모호 제약·누락 예시·스토리 보강·테스트 추가처럼 새 집필이 필요한 보완은 2단계로 되돌아가** 해당 부분만
   다시 쓴다. 재채점하고 라운드를 올린다. **PASS면 종료**. 3라운드를 채워도 FAIL이면 **멈추고** 남은 미흡과
   자동 보완이 어려운 이유를 보고한다 — 기준을 낮춰 임의 통과시키지 않는다.
4. 호출 맥락:
   - **사용자 직접 호출**: 라운드 1 결과를 보여준 뒤 "(a) PASS까지 자동 보완 / (b) 항목 골라 보완 / (c) 여기까지"를 묻는다.
   - **파이프라인 안에서**: 루프를 자동으로 돌리되 라운드마다 무엇을 고쳤는지 남기고, PASS 전에는 발행으로 넘어가지 않는다.

## 4단계 — 발행 (publishing)

> 블로그 `publish-post` 대응 단계. 문제는 별도 콘텐츠 컬렉션 발행 대신 **빌드·테스트 확정 + (해당 시) 목록
> 등록**으로 발행을 마친다(블로그가 콘텐츠 컬렉션 자동 탐색인 것과 동일한 정신).

PASS한 뒤에만 진행한다.

1. **최종 점검**: 문제 문서가 `src/$ARGUMENTS-problem.md`(또는 `.mdx`)에 있고, 테스트가
   `src/$ARGUMENTS.test.ts`에 있다. `.mdx`면 import 경로·JSX가 올바른지 확인한다.
2. **빌드·테스트 확인**: `bun test src/$ARGUMENTS.test.ts`가 통과하는지(구현이 있을 때), `.mdx`면
   `bun run build`(또는 타입 체크)가 통과하는지 마지막으로 확인한다.
3. **목록 등록(해당 시)**: 문제 목록/인덱스를 생성하는 도구가 있으면 실행해 새 문제가 등록되게 한다.
4. **완료 보고**: 생성된 문제 문서·테스트 경로, 테스트 통과 여부, 품질 게이트 최종 판정(PASS/라운드 수)을
   한 번에 보고한다.

## 참조 파일

- `checklist.md` — 테스트 커버리지 기준(바운더리·성능·엣지)
- `assets/PROBLEM_QUALITY_CHECKLIST.md` — 3단계 품질 게이트 루브릭(MUST/SHOULD/IF-APPLICABLE)
- `scripts/review-problem.sh` — 2.3 외부 검토(codex/agy: 모호·모순·예시정확성·힌트누출)

## 주의

- **구성 항목·금기는 바꾸지 않는다.** 워크플로우(4단계)만 재정의했고, 문제 골격(함수 인터페이스/제약 조건/
  문제 상세/예시)과 금기(파라미터 언더바 prefix 금지, **풀이 힌트 미포함**)는 그대로다.
- **스토리는 힌트가 아니다.** 동기·맥락만 준다 — 알고리즘·자료구조·접근법을 암시하면 풀이 누출이다.
- **자동 통과 금지**: 품질 게이트 3라운드를 다 써도 미통과면 그대로 보고한다.
- **외부 검토 미통과(둘 다 실패)** 상태를 "완성"으로 보고하지 않는다.
- 글과 테스트는 **같은 계약**을 말한다 — 예시 출력과 테스트 기대값이 어긋나면 안 된다.
- `.mdx` 작성 후에는 빌드 통과를 확인한다(JSX 문법 오류 1건이 빌드 전체를 깰 수 있다).
